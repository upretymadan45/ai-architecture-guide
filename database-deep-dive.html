Performance 01
Indexing — The #1 Performance Lever
An index is a sorted data structure that lets the DB find rows without scanning the whole table — turning O(n) scans into O(log n) lookups.

B-Tree index (default)

  • Balanced tree, sorted keys
  • Great for equality & range queries
  • Powers WHERE, ORDER BY, joins
  • The default in Postgres, MySQL, SQL Server

LSM-Tree index

  • Write-optimized (sequential writes, compaction)
  • Used by Cassandra, RocksDB, many NoSQL stores
  • Fast ingest; reads may touch multiple levels
  • Trades read amplification for write throughput
Index typeUse for
CompositeMulti-column filters — order matters (leftmost-prefix rule)
CoveringIndex includes all queried columns → no table lookup
UniqueEnforce uniqueness + speed lookups
Partial / filteredIndex only rows matching a condition (e.g. WHERE active = true)
HashEquality-only, very fast point lookups
GIN / invertedFull-text search, arrays, JSON
⚖️
Indexes aren't free. They speed reads but slow writes (every insert/update maintains the index) and consume storage. Index the columns you filter, join, and sort on — not every column. The leftmost-prefix rule: a composite index on (a, b, c) helps queries on a, a,b, or a,b,c — but not on b alone.
🚫
Index killers: wrapping a column in a function (WHERE UPPER(name) = …), leading wildcards (LIKE '%foo'), implicit type casts, or OR across different columns can all force a full scan despite an index existing. Check the query plan to confirm the index is actually used.
Correctness 02
Transactions & ACID
A transaction groups operations into an all-or-nothing unit. ACID is the guarantee that keeps your data consistent under concurrency and failure.
⚛️
Atomicity
All operations succeed or none do. A failed transfer never debits without crediting — it rolls back entirely.
✅
Consistency
Every transaction moves the DB from one valid state to another — constraints, triggers, and rules always hold.
🔒
Isolation
Concurrent transactions don't interfere — results look as if they ran one at a time (degree set by isolation level).
💾
Durability
Once committed, data survives crashes — guaranteed by the write-ahead log (WAL) flushed to durable storage.
📝
The write-ahead log (WAL) is the unsung hero. Changes are written to an append-only log before being applied to data files. On crash, the DB replays the log to recover — giving both durability and atomicity. It's also the foundation of replication (ship the log to replicas).
Concurrency 03
Isolation Levels & Anomalies
Stronger isolation = more correctness but less concurrency. Each level prevents specific anomalies — know what you're trading.
LevelDirty ReadNon-Repeatable ReadPhantom Read
Read Uncommitted❌ Possible❌ Possible❌ Possible
Read Committed✓ Prevented❌ Possible❌ Possible
Repeatable Read✓ Prevented✓ Prevented❌ Possible*
Serializable✓ Prevented✓ Prevented✓ Prevented

* Postgres's Repeatable Read (snapshot isolation) also prevents phantoms in practice.

💧
Dirty Read
Reading another transaction's uncommitted changes that may roll back.
🔁
Non-Repeatable Read
Re-reading the same row returns a different value because another txn committed an update in between.
👻
Phantom Read
Re-running the same query returns new rows because another txn inserted matching records.
⚔️
Lost Update
Two txns read-modify-write the same row; one overwrites the other's change. Use optimistic/pessimistic locking.
🎚️
Read Committed is the common default (Postgres, Oracle, SQL Server). Bump to Serializable only where correctness demands it (financial invariants) — it's safest but can abort transactions under contention, so the app must retry. Pick the weakest level that still preserves your invariants.
Internals 04
Locking & MVCC
How databases let many transactions run concurrently without corrupting each other — two competing philosophies.

Pessimistic locking

  • Lock rows before modifying (SELECT ... FOR UPDATE)
  • Others wait — no conflicts, but contention
  • Good for high-conflict, short transactions
  • Risk: deadlocks & reduced throughput

MVCC (optimistic, snapshot)

  • Readers see a consistent snapshot — never block writers
  • Writers create new row versions; old ones kept for readers
  • Powers Postgres, MySQL/InnoDB, Oracle
  • Cost: version cleanup (vacuum) & write conflicts
🧬
MVCC's key insight: readers don't block writers and writers don't block readers. Each transaction sees a snapshot of the data as of its start. This dramatically improves concurrency vs pure locking — at the cost of storing multiple row versions and periodic garbage collection (Postgres VACUUM).
🔐
Deadlocks happen when txn A holds lock 1 and wants lock 2 while txn B holds lock 2 and wants lock 1. The DB detects the cycle and kills one (a victim) — the app must catch & retry. Prevent them: acquire locks in a consistent order, keep transactions short, and touch the fewest rows possible.
Tuning 05
Query Optimization & Execution Plans
The query planner turns your SQL into a physical plan. Reading that plan is the single most useful database debugging skill.
EXPLAIN ANALYZE — see what the DB actually does
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42 AND status = 'open'; -- Look for: Seq Scan (bad on big tables) vs Index Scan (good), -- estimated vs actual rows (stale stats?), and the costliest node.
Plan nodeMeaningVerdict
Seq ScanRead every rowFine on small tables, red flag on big ones
Index ScanUse an index to find rowsUsually what you want
Index Only ScanAnswer entirely from a covering indexBest — no table access
Nested LoopJoin by loopingGreat for small result sets
Hash JoinBuild a hash, probe itGood for large unsorted joins
Merge JoinJoin two sorted inputsGood when inputs already sorted
🔧
Optimization checklist: add the right index, avoid SELECT * (fetch only needed columns → enables covering indexes), eliminate N+1 queries (batch/join instead), keep statistics fresh (ANALYZE), and watch for the bottleneck node with the highest actual time. Measure before and after — never optimize blind.
Modeling 06
Normalization vs Denormalization
Normalize for write integrity; denormalize for read speed. The right answer depends on your access patterns.

Normalization (3NF)

  • Each fact stored once → no update anomalies
  • Smaller tables, strong integrity
  • Great for write-heavy OLTP systems
  • Cost: more joins to assemble data

Denormalization

  • Duplicate data to avoid joins → fast reads
  • Great for read-heavy / analytics / feeds
  • Cost: redundancy, update complexity, drift risk
  • Often a cache or materialized view, not the source of truth
⚖️
Default to normalized, denormalize deliberately. Start with a clean 3NF model for correctness, then denormalize specific hot paths (materialized views, precomputed aggregates, read models) when profiling proves joins are the bottleneck. Document the redundancy and how it stays in sync — uncontrolled duplication is how data goes wrong.
🧱
OLTP vs OLAP shapes the model. Transactional systems (OLTP) favor normalized rows for fast writes & point lookups. Analytical systems (OLAP / data warehouses) favor denormalized star/snowflake schemas and columnar storage for fast aggregations over huge scans. Use the right store for each workload.
Scale 07
Partitioning, Replication & Pooling
When one machine isn't enough, you spread the load — each technique with its own trade-offs.
📑
Partitioning
Split one table by range/hash/list (e.g. by date) so queries touch fewer rows. Same DB, smaller working set.
🧩
Sharding
Horizontal partitioning across machines by a shard key. Scales writes & storage; cross-shard joins get hard.
📋
Replication
Copy data to replicas. Read replicas scale reads & add HA; async replication means replica lag (stale reads).
🔌
Connection Pooling
Reuse a fixed set of DB connections. Connections are expensive — a pool (PgBouncer, built-in) prevents exhaustion.
🔑
The shard key is the most important decision. A bad key creates hotspots (one shard does all the work) and makes cross-shard queries painful. Choose a key with high cardinality and even distribution that matches your dominant access pattern. Resharding later is expensive — get it right early, or avoid sharding until you truly need it.
⏱️
Replica lag breaks read-your-writes. With async replication, a user may write to the primary then read a stale value from a replica. Fixes: route critical reads to the primary, use synchronous replication for must-be-fresh data, or pin a session to the primary briefly after a write. Scaling reads with replicas is easy; scaling writes (sharding) is the genuinely hard problem.
Choice 08
SQL vs NoSQL — Picking the Right Store
"NoSQL" is many things. Match the data model and access pattern to the store, not the hype.
Store typeModelBest forExamples
RelationalTables + joins, ACIDStructured data, transactions, complex queriesPostgres, MySQL, SQL Server
DocumentJSON documentsFlexible/evolving schema, aggregates per entityMongoDB, CosmosDB
Key-ValueHash mapCaching, sessions, ultra-fast lookupsRedis, DynamoDB
Wide-ColumnSparse columns by keyMassive write throughput, time-seriesCassandra, ScyllaDB
GraphNodes + edgesRelationship-heavy (social, fraud, recommendations)Neo4j, Neptune
SearchInverted indexFull-text search, log analyticsElasticsearch, OpenSearch
🎯
Relational is the right default for most apps. Reach for NoSQL when you have a specific need it serves better: extreme scale, flexible schema, a non-tabular data shape, or a particular access pattern. Modern Postgres handles JSON, full-text search, and huge scale — often you don't need a separate NoSQL store at all. Polyglot persistence (different stores for different jobs) is common at scale.
📐
NoSQL design is access-pattern-first. Relational modeling normalizes around the data; NoSQL modeling (especially DynamoDB/Cassandra) starts from the queries you'll run and shapes the data to serve them cheaply — often denormalizing heavily. Know your queries before you design the schema.
