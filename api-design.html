Foundation 01
REST Principles & Resource Modeling
REST treats everything as a resource identified by a URL, manipulated with standard HTTP verbs. Get the resource model right and the rest follows.

Good resource design

  • Nouns, not verbs: /orders/42
  • Plural collections: /users, /users/7
  • Nest for relationships: /users/7/orders
  • Lowercase, hyphenated, no file extensions
  • Filter/sort via query: ?status=open&sort=-date

Anti-patterns

  • Verbs in paths: /getUser?id=7
  • Actions as resources: /createOrderNow
  • Inconsistent singular/plural mixing
  • Tunneling everything through POST
  • Leaking DB tables as the API shape
📐
Richardson Maturity Model grades REST adoption: L0 single endpoint (RPC over HTTP) → L1 separate resources → L2 proper HTTP verbs & status codes (where most good APIs live) → L3 HATEOAS (responses include links to next actions). Aim for solid L2; L3 is valuable but rarely fully adopted.
🧩
Design for the consumer, not the database. The API is a product. Model resources around what clients need to do, hide internal storage details, and keep representations stable. Sometimes an "action" genuinely isn't CRUD — model it as a sub-resource (POST /orders/42/cancellation) rather than forcing a verb.
Semantics 02
HTTP Methods & Status Codes
Use the verbs and codes the web already understands — clients, caches, and proxies all rely on them.
MethodPurposeSafe?Idempotent?
GETRead a resourceYesYes
POSTCreate / non-idempotent actionNoNo
PUTReplace a resource fullyNoYes
PATCHPartial updateNoNo*
DELETERemove a resourceNoYes

* PATCH can be made idempotent depending on semantics. Safe = no state change; idempotent = same result if repeated.

CodeMeaningWhen

400Bad RequestMalformed / validation failed

⚠️
Don't return 200 OK with an error in the body. It breaks clients, monitoring, and caches that trust the status line. Use 4xx for client mistakes (don't retry blindly) and 5xx for server faults (safe to retry with backoff). The status code is part of your contract.
Evolution 03
Versioning & Backward Compatibility
APIs are forever — once a client depends on yours, you can't break it. Plan to evolve without breaking.
StrategyExampleNotes
URI path/v1/usersMost common, explicit, cache-friendly
HeaderAccept: application/vnd.api.v2+jsonClean URLs; less discoverable
Query param/users?version=2Simple; easy to forget

Non-breaking changes (safe)

  • Add a new optional field
  • Add a new endpoint
  • Add a new optional query param
  • Add a new enum value (if clients tolerate)

Breaking changes (need a version)

  • Remove or rename a field
  • Change a field's type or meaning
  • Make an optional field required
  • Change error/status semantics
🔄
Tolerant reader + additive changes avoid most version churn: clients ignore unknown fields, servers only add. Reserve new major versions for true breaking changes, support old versions during a deprecation window with clear timelines & Sunset headers, and communicate changes loudly.
Robustness 04
Pagination, Filtering & Idempotency
The practical patterns that make an API scalable and safe to call over flaky networks.

Offset pagination

  • ?limit=20&offset=40
  • Simple, jump to any page
  • Slow & inconsistent on large/changing data

Cursor / keyset pagination

  • ?limit=20&cursor=eyJpZCI6NDB9
  • Stable under inserts, fast at scale
  • Can't jump to arbitrary page — best for feeds
IdempotencyMake retries safe

Networks fail after the server processed a request but before the client got the response. Without protection, the client retries and you charge the card twice.

Idempotency key — dedupe POST retries
POST /payments Idempotency-Key: 3f8a-9c21-bd47 # Server: first time → process + store result under the key. # Retry with same key → return the stored result, don't re-charge.
🔁
Idempotency is essential for any "write" that costs something. GET, PUT, DELETE are naturally idempotent; POST needs an idempotency key. Pair it with optimistic concurrency (ETag + If-Match → 409 Conflict) to handle simultaneous edits safely.
🚦
Rate limiting protects you from abuse & runaway clients. Communicate limits via headers (X-RateLimit-Remaining, Retry-After) and return 429 when exceeded. Common algorithms: token bucket (allows bursts) and sliding window (smoother). Good clients back off; bad ones get throttled.
Choice 05
REST vs gRPC vs GraphQL
There's no "best" — each fits a different shape of problem. Pick deliberately.
RESTgRPCGraphQL
TransportHTTP/1.1+ JSONHTTP/2 + ProtobufHTTP + JSON
ContractOpenAPI (optional).proto (strict)Schema (strict)
ShapeFixed per endpointFixed RPC methodsClient picks fields
StreamingLimitedFirst-class (bi-di)Subscriptions
Best forPublic/web APIsInternal microservicesAggregating many sources / rich clients
🌍
Choose REST
Public APIs, broad client support, cacheable reads, simplicity. The safe default for most web-facing APIs.
⚡
Choose gRPC
Low-latency internal service-to-service calls, strong typed contracts, streaming, polyglot codegen.
🎯
Choose GraphQL
Diverse clients needing different data shapes; avoids over/under-fetching by letting clients query exactly what they need.
⚖️
Trade-offs to weigh: GraphQL solves over-fetching but adds caching, query-cost, and N+1 complexity on the server. gRPC is fast & typed but not browser-native (needs a proxy) and harder to debug by hand. REST is universally understood but can be chatty. Match the tool to the consumer.
Reliability 06
Error Handling & Resilience
Errors are part of the contract. Make them consistent, machine-readable, and actionable.
RFC 9457 — Problem Details, a standard error shape
{ "type": "https://api.example.com/errors/insufficient-funds", "title": "Insufficient funds", "status": 409, "detail": "Balance 30.00 is less than the 50.00 charge.", "instance": "/accounts/123/charges", "traceId": "abc-123-def" }

Good error design

  • Consistent shape across all endpoints
  • Correct status code + stable error code
  • Human message + machine code + trace id
  • Field-level details for validation
  • Never leak stack traces / internals

Client-side resilience

  • Retry 5xx/429 with exponential backoff + jitter
  • Respect Retry-After
  • Circuit breaker to stop hammering a dead service
  • Timeouts on every call — never wait forever
🛟
Design errors for the caller. A stable error code lets clients branch programmatically without parsing prose. Include a correlation/trace id so support can find the exact request in logs. Only retry idempotent or idempotency-key'd operations — retrying a non-idempotent POST can double-charge.
Security 07
API Security Essentials
An exposed API is an attack surface. These are the non-negotiables.
🪪
AuthN vs AuthZ
Authentication = who you are (OAuth2/OIDC, JWT). Authorization = what you can do (scopes, roles). Check both on every request.
🎫
OAuth2 / OIDC
Delegated access via tokens — don't share passwords. Short-lived access tokens + refresh tokens. Validate signature, issuer, audience, expiry.
🔒
Transport security
TLS everywhere (HTTPS only), HSTS. Never send tokens/keys over plaintext or in URLs (they land in logs).
🧼
Input validation
Validate & sanitize all input at the boundary. Parameterized queries (no SQL injection), strict schemas, size limits.
🛡️
OWASP API Security Top 10 hotspots: Broken Object-Level Authorization (BOLA) — always check the caller owns the resource id they request (don't trust /orders/42 just because they're logged in); Broken Authentication; Excessive Data Exposure (don't dump the whole DB object — return a DTO); and lack of rate limiting. Authorize on the server, every time, per object.
Process 08
Contract-First & OpenAPI
Design the contract before the code. The spec becomes the single source of truth that humans and machines share.
1
Design the spec (OpenAPI/Swagger for REST, .proto for gRPC, SDL for GraphQL) before implementing.
2
Review with consumers — frontend, partners, other teams — so the contract fits real needs.
3
Generate server stubs, client SDKs, & mock servers from the spec — no drift between docs & code.
4
Validate in CI — contract tests & lint catch breaking changes before they ship.
5
Publish living docs from the same spec so documentation is always accurate.
📜
The contract is the product boundary. Contract-first decouples teams (frontend builds against mocks while backend implements), prevents drift between docs and reality, and turns "did you break the API?" into an automated CI check. Consumer-driven contract tests (e.g. Pact) verify the provider still satisfies what each consumer actually relies on.
✨
The hallmarks of a great API: consistent (naming, errors, pagination everywhere the same), predictable (least surprise), well-documented, evolvable without breaking, secure by default, and easy to do the right thing. If a new developer can guess your next endpoint correctly, you've designed it well.
