Foundation 01
DevOps Culture — Not a Tool, a Way of Working
DevOps removes the wall between "dev" and "ops": shared ownership of building and running software, with fast feedback and automation everywhere.
🤝
Shared ownership
"You build it, you run it." Teams own their service end-to-end — code, deploy, on-call. Removes hand-off friction & blame.
🔄
Fast feedback
Short loops: automated tests, quick deploys, real-time monitoring. Find & fix problems in minutes, not weeks.
🤖
Automate everything
Builds, tests, deploys, infra. Humans for judgement, machines for repetition — fewer errors, more speed.
📉
Blameless culture
Failures are system problems, not people problems. Blameless post-mortems → learning & prevention, not finger-pointing.
📊
The DORA metrics measure delivery performance: Deployment Frequency, Lead Time for Changes, Change Failure Rate, and Mean Time to Recovery (MTTR). Elite teams deploy often, with short lead times, low failure rates, and fast recovery — and research shows these correlate with business success.
Pipeline 02
CI/CD — From Commit to Production
Continuous Integration merges & verifies changes constantly; Continuous Delivery/Deployment automates the path to production.

Continuous Integration (CI)

  • Merge to main frequently (small changes)
  • Every push triggers build + automated tests
  • Catch integration bugs early, keep main green
  • Static analysis, linting, security scans

Continuous Delivery / Deployment (CD)

  • Delivery: always release-ready; deploy is one click
  • Deployment: every green build auto-ships to prod
  • Automated, repeatable, auditable releases
  • Fast rollback if something goes wrong
A typical pipeline
flowchart LR
  C["Commit / PR"] --> B["Build"]
  B --> T["Unit + Integration\nTests"]
  T --> SA["Static Analysis\n+ Security Scan"]
  SA --> A["Artifact / Image\n(versioned)"]
  A --> ST["Deploy → Staging"]
  ST --> AT["Acceptance /\nSmoke Tests"]
  AT --> PR["Deploy → Prod\n(canary / blue-green)"]
  PR --> M["Monitor + Rollback"]
⚙️
Pipeline principles: build once, promote the same artifact through environments (never rebuild per stage); fail fast (cheap checks first); keep it fast (parallelize, cache dependencies); make it the only path to prod. Tools: GitHub Actions, GitLab CI, Azure DevOps, Jenkins.
GitHub Actions — minimal CI workflow
name: ci on: [push, pull_request] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-dotnet@v4 with: { dotnet-version: '8.0' } - run: dotnet restore - run: dotnet build --no-restore -c Release - run: dotnet test --no-build -c Release # fail fast on red tests
Packaging 03
Containers — "Works on My Machine," Solved
A container packages an app with its dependencies into one portable, isolated unit that runs identically everywhere.

Containers (Docker)

  • Share the host OS kernel — lightweight, fast start
  • MBs in size; seconds to launch
  • Consistent across dev → CI → prod
  • Ideal unit for microservices & scaling

Virtual Machines

  • Full guest OS per VM — strong isolation
  • GBs in size; minutes to boot
  • Heavier, but stronger security boundary
  • Better for mixed OS / strict isolation needs
Multi-stage Dockerfile — small, secure image
# Build stage — has the full SDK FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build WORKDIR /src COPY . . RUN dotnet publish -c Release -o /app # Runtime stage — tiny, no build tools shipped FROM mcr.microsoft.com/dotnet/aspnet:8.0 WORKDIR /app COPY --from=build /app . USER app # run as non-root ENTRYPOINT ["dotnet", "MyApi.dll"]
🛡️
Image best practices: use multi-stage builds (don't ship the SDK), pin base-image versions, run as a non-root user, keep images minimal (smaller = faster pulls + smaller attack surface), scan for vulnerabilities, and never bake secrets into layers. .dockerignore to keep the context lean.
Orchestration 04
Kubernetes — Orchestrating Containers at Scale
When you have many containers across many machines, you need an orchestrator to schedule, heal, scale, and network them. K8s is the standard.
📦
Pod
The smallest deployable unit — one or more containers sharing network/storage. Usually one app container per pod.
🔁
Deployment
Declares desired state (N replicas of an image). K8s reconciles reality to match — self-healing & rolling updates.
🌐
Service
A stable network endpoint & load balancer over a changing set of pods. Decouples callers from pod IPs.
🚪
Ingress
HTTP(S) routing into the cluster — host/path rules, TLS termination — to the right Service.
📈
HPA (Autoscaler)
Horizontal Pod Autoscaler adds/removes pods based on CPU/memory/custom metrics.
🔐
ConfigMap / Secret
Inject configuration & secrets into pods without rebuilding images. Keep config out of code.
🧭
The declarative model is the big idea: you describe the desired state in YAML; the control loop continuously reconciles actual → desired. A pod dies? It's recreated. A node fails? Pods reschedule. You declare "what", K8s figures out "how" and keeps it true.
⚖️
Kubernetes is powerful but heavy. It's a steep operational investment — don't reach for it for a single small service. Managed options (EKS, AKS, GKE) reduce the burden; serverless containers (Cloud Run, ECS Fargate, Container Apps) or a PaaS may be the simpler, cheaper fit until scale demands K8s.
Automation 05
Infrastructure as Code (IaC)
Define servers, networks, and services in version-controlled files instead of clicking in a console. Reproducible, reviewable, automatable.

Why IaC

  • Reproducible environments (dev = staging = prod)
  • Versioned, peer-reviewed, auditable changes
  • No "snowflake" servers or config drift
  • Disaster recovery = re-run the code
  • Self-documenting infrastructure

Declarative vs Imperative

  • Declarative (Terraform, Bicep, CloudFormation): describe the end state, tool computes the diff
  • Imperative (scripts, Ansible tasks): list the steps to run
  • Declarative is preferred — idempotent & drift-aware
  • GitOps: Git is the source of truth, auto-applied
Terraform — declarative, idempotent
resource "aws_s3_bucket" "assets" { bucket = "my-app-assets" tags = { Environment = "prod" } } # `terraform plan` shows the diff; `apply` makes reality match. # Run it twice → no change (idempotent).
📌
IaC best practices: keep state secure & locked (remote backend), modularize reusable components, never edit infra by hand (it causes drift), review plans before apply, and treat infra code with the same rigor as app code — tests, CI, peer review.
Release 06
Deployment Strategies — Ship Without Downtime
How you roll out a new version determines your blast radius when something goes wrong.
StrategyHow it worksTrade-off
RollingReplace instances gradually, a few at a timeNo downtime, simple; mixed versions briefly
Blue-GreenTwo identical envs; switch traffic all at onceInstant rollback; doubles infra cost
CanaryRelease to a small % of users, watch metrics, ramp upLimits blast radius; needs good monitoring & routing
Feature FlagsDeploy code dark, toggle features on per cohortDecouples deploy from release; flag debt to manage
ShadowMirror real traffic to the new version without serving itSafe real-world test; extra infra & complexity
🐤
Canary + feature flags is the modern default. Deploy the artifact safely (canary watches error rate & latency, auto-rolls-back on regression), and control feature exposure separately with flags — so "deploy" and "release" become independent, low-risk events. Roll back a feature instantly without a redeploy.
🗄️
Database migrations are the hard part of zero-downtime. Use the expand/contract pattern: add new columns/tables (backward-compatible) → deploy code that writes both → backfill → switch reads → remove the old. Never make a breaking schema change in the same release that depends on it.
Operations 07
Observability & SRE
Once it's running, you have to see it. Observability + Site Reliability Engineering keep production healthy and improve it over time.
📊
Metrics
Numeric time-series — RPS, latency percentiles, error rate, saturation. Cheap, great for dashboards/alerts. Prometheus + Grafana.
📝
Logs
Structured event records for detail. Centralize & index (ELK/Loki). Expensive at volume — sample wisely.
🧵
Traces
Follow one request across services via a trace id. Pinpoints the slow/broken hop. OpenTelemetry, Jaeger.
🔔
Alerting
Alert on user-facing symptoms (SLO breaches), not noise. Every alert: actionable, owned, runbooked.
🎯
SRE core ideas: define SLIs (measured signals), set SLOs (targets, e.g. 99.9% of requests < 200 ms), and spend the error budget deliberately — if you're within budget, ship faster; if you've burnt it, focus on reliability. Toil (manual, repetitive work) is the enemy — automate it away.
🚦
Four Golden Signals: Latency, Traffic, Errors, Saturation. Health checks (liveness + readiness) let the orchestrator route around sick instances. Propagate a correlation ID through every hop so a single request can be reconstructed across logs, metrics, and traces.
Security 08
DevSecOps — Shift Security Left
Bake security into the pipeline from the start instead of bolting it on at the end. Automated, continuous, everyone's job.
1
SAST — static analysis scans your source for vulnerabilities on every commit.
2
SCA / dependency scanning — flag known-vulnerable libraries (dotnet list package --vulnerable, Dependabot, Snyk).
3
Container & image scanning — check base images & layers for CVEs before they ship.
4
Secret scanning — block commits that contain keys/tokens; store secrets in a vault.
5
DAST — dynamic testing against the running app in staging.
6
Policy as code — enforce guardrails (no public buckets, encryption on) automatically in CI.
🔐
Core principles: least privilege for pipelines & runtime, secrets in a managed vault (never in code/images), sign & verify artifacts (supply-chain integrity, SLSA), immutable infrastructure (replace, don't patch in place), and automate compliance. Security that's automated is security that actually happens — manual checks get skipped under deadline pressure.
