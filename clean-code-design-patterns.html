Topic 01
Clean Code — Optimize for Reading
Code is read far more often than it's written. Clean code is not about cleverness — it's about clarity, so the next person (often you in six months) understands it instantly.
🏷️
Meaningful names
Reveal intent. elapsedDays beats d. Pronounceable, searchable, no encodings. A good name removes the need for a comment.
🔬
Small functions
Do one thing, one level of abstraction, few arguments. If you need "and" to describe it, split it.
💬
Comments explain "why"
Good code is self-documenting for "what"; comments justify the non-obvious "why". Delete commented-out code.
🧯
Fail fast, guard early
Validate at boundaries, return early, avoid deep nesting. Flat code with guard clauses reads top-to-bottom.
Unclear — what does this even do?
public List<int> gd(List<int> l) { var r = new List<int>(); foreach (var x in l) if (x % 2 == 0) r.Add(x); // magic, cryptic names return r; }
Intent is obvious
public List<int> GetEvenNumbers(List<int> numbers) => numbers.Where(n => IsEven(n)).ToList(); private bool IsEven(int n) => n % 2 == 0;
🎯
The Boy Scout Rule: leave the code cleaner than you found it. Small, continuous improvements compound; big-bang rewrites rarely happen. Clean code is a habit, not a project.
Topic 02
SOLID Principles
Five principles for object-oriented design that keep code flexible, testable, and resistant to rot.
PrincipleOne-linerViolation smell
S — Single ResponsibilityA class has one reason to changeGod class doing persistence + validation + formatting
O — Open/ClosedOpen to extension, closed to modificationA growing switch you edit for every new type
L — Liskov SubstitutionSubtypes must be usable as their baseOverride that throws NotSupportedException
I — Interface SegregationMany focused interfaces > one fat oneImplementing methods you leave empty / unsupported
D — Dependency InversionDepend on abstractions, not concretesnew SqlRepository() hard-wired inside logic
OCP violation — edit the class for every new shape
public double Area(object shape) { if (shape is Circle c) return Math.PI * c.R * c.R; if (shape is Square s) return s.Side * s.Side; // add another 'if' for every new shape… forever throw new NotSupportedException(); }
OCP — extend by adding a class, not editing
public interface IShape { double Area(); } public record Circle(double R) : IShape { public double Area() => Math.PI * R * R; } public record Square(double Side) : IShape { public double Area() => Side * Side; } // New shapes plug in — no existing code changes.
⚖️
SOLID is a guide, not a religion. Each principle reduces coupling but adds indirection. Applied blindly you get a maze of one-method interfaces. Apply them where change is likely; leave simple, stable code simple.
Topic 03
DRY, KISS, YAGNI & Friends
The maxims that keep designs honest. Most over-engineering comes from forgetting the last two.
♻️
DRY
Don't Repeat Yourself — every piece of knowledge has one authoritative home. But beware: duplicated code isn't always duplicated knowledge.
💋
KISS
Keep It Simple — the simplest thing that works. Complexity must earn its place.
🔮
YAGNI
You Aren't Gonna Need It — don't build for imagined future requirements. Add flexibility when the need is real.
🧩
Composition over inheritance
Prefer assembling behavior from parts over deep class hierarchies. Inheritance is rigid; composition flexes.
🔗
Low coupling, high cohesion
Modules should depend on little (loose coupling) and group related things (high cohesion). The core of good structure.
🚪
Law of Demeter
Talk only to your friends — a.B.C.do() chains couple you to internals. Ask, don't reach through.
⚠️
The DRY trap: over-aggressive DRY creates the wrong abstraction — coupling unrelated code that merely looked similar. "A little copying is better than a little dependency." Wait until you see the pattern three times before extracting it (the Rule of Three).
Topic 04
Code Smells — Symptoms of Decay
Smells aren't bugs; they're hints that the design will hurt to change. Learn to notice them early.
SmellWhat you seeTypical fix
Long methodHundreds of lines, many responsibilitiesExtract method, decompose
God classOne class that knows/does everythingExtract class, apply SRP
Long parameter list5+ args, easy to mis-orderParameter object, builder
Feature envyA method uses another class's data more than its ownMove method to where the data lives
Shotgun surgeryOne change forces edits in many placesConsolidate responsibility
Primitive obsessionStrings/ints for domain conceptsValue objects (Money, Email)
Magic numbersUnexplained literalsNamed constants / enums
Deep nestingArrow-shaped if pyramidsGuard clauses, early return
🧭
Smells guide, tests protect. Don't refactor on a smell alone — confirm it's painful to change, ensure tests cover the behavior, then improve in small, verified steps. Refactoring without tests is just rewriting and praying.
Patterns 05
Creational Patterns — How Objects Are Made
Decouple your code from the messy details of construction.
🏭
Factory Method
Defer instantiation to a method/subclass so callers depend on an interface, not a concrete type. e.g. IHttpClientFactory.
🏢
Abstract Factory
Create families of related objects (e.g. a UI toolkit per OS) without naming concretes.
🔨
Builder
Construct complex objects step-by-step with a fluent API. Great for many optional parameters.
1️⃣
Singleton
One shared instance. Useful for config/caches — but a global, so prefer DI-managed singletons over the classic static.
Builder — readable construction of a complex object
var report = new ReportBuilder() .WithTitle("Q2 Sales") .AddSection("Revenue", revenueData) .AddSection("Costs", costData) .WithFooter("Confidential") .Build();
🚫
Singleton caution: the classic static singleton is global mutable state — it hides dependencies, breaks testability, and risks thread-safety bugs. In modern apps, register a single instance in the DI container instead (lifetime = singleton) so it stays injectable and mockable.
Patterns 06
Structural Patterns — How Objects Compose
Assemble objects and classes into larger structures while keeping them flexible.
🔌
Adapter
Wrap an incompatible interface to make it fit. The classic "translate this third-party API to ours".
🎀
Decorator
Wrap an object to add behavior (logging, caching, retry) without changing it. Composable & SRP-friendly.
🚪
Facade
A simple front over a complex subsystem. One clean entry point hides the messy internals.
🛡️
Proxy
A stand-in that controls access — lazy loading, caching, access control, remote calls.
Decorator — add caching without touching the service
public class CachingProductService : IProductService { private readonly IProductService _inner; // the real service private readonly IMemoryCache _cache; public async Task<Product> GetAsync(int id) => await _cache.GetOrCreateAsync(id, _ => _inner.GetAsync(id)); // wraps }
Patterns 07
Behavioral Patterns — How Objects Collaborate
Define how objects interact and distribute responsibility.
🧭
Strategy
Swap interchangeable algorithms behind an interface (sorting, pricing, payment). Pick at runtime.
👀
Observer
Publish/subscribe — observers react to a subject's changes. The basis of events & reactive UIs.
📬
Command
Encapsulate a request as an object — enables queues, undo/redo, logging. Core of CQRS/MediatR.
🔁
Template Method
A base defines the skeleton; subclasses fill in steps. Shared flow, varied details.
🚦
State
Behavior changes with internal state — replace sprawling state if/switch with state objects.
🔗
Chain of Responsibility
Pass a request along a chain of handlers until one handles it. e.g. middleware pipelines.
Strategy — pluggable pricing
public interface IPricingStrategy { decimal Price(Cart c); } public class RegularPricing : IPricingStrategy { ... } public class BlackFridayPricing : IPricingStrategy { ... } // Inject the right strategy; the checkout code never changes. var total = _strategy.Price(cart);
🧠
Patterns are a vocabulary. Their biggest value is shared language: saying "let's decorate the repository with caching" communicates a whole design instantly. Recognize them in frameworks you already use — ASP.NET middleware (chain of responsibility), IObservable (observer), MediatR (command/mediator).
Practice 08
Refactoring — Changing Structure, Not Behavior
Refactoring improves the design of existing code without changing what it does — in small, test-protected steps.
1
Ensure tests exist. A green test suite is your safety net — it proves behavior is preserved.
2
Make one small change. Extract a method, rename a variable, introduce a parameter object.
3
Run the tests. Green? Commit. Red? Revert — the step was too big or wrong.
4
Repeat. Many tiny safe steps beat one risky rewrite.

Common refactorings

  • Extract Method / Extract Class
  • Rename for clarity
  • Replace magic number with constant
  • Introduce parameter object
  • Replace conditional with polymorphism
  • Guard clauses to flatten nesting

When to refactor

  • Before adding a feature to messy code (clean, then add)
  • When you keep hitting the same friction
  • During code review feedback
  • As part of the Boy Scout Rule, opportunistically
  • Not a separate "refactor sprint" with no goal
🛑
Refactoring vs rewriting: refactoring preserves behavior in small steps; a rewrite throws it away and risks reintroducing every old bug. Prefer the strangler-fig approach — incrementally replace parts of a system while it keeps running.
Judgement 09
When NOT to Use a Pattern
The senior skill isn't knowing patterns — it's knowing when the cost of abstraction outweighs the benefit.

Over-engineering signs

  • An interface with exactly one implementation, forever
  • A factory that just calls new
  • A repository that only forwards to EF Core
  • Layers of indirection to read a config value
  • Patterns applied to satisfy a checklist, not a need

Earn the abstraction

  • Add it when a second real variation appears
  • Optimize for deletion & change, not prediction
  • Prefer the simplest design that meets the requirement
  • Make the code obvious before making it clever
  • Measure complexity cost vs flexibility gained
🧠
The senior verdict: "make it work, make it right, make it fast — in that order." Every abstraction is a bet that change will come. Good engineers place those bets where change is likely, and keep everything else boring, direct, and easy to delete. Simplicity is a feature.
