Foundation 01
Big-O — Measuring How Things Grow
Big-O describes how runtime/space scales as input grows. It's about the growth rate, not the stopwatch — constants drop, the dominant term wins.
NotationNameExample
O(1)ConstantHash lookup, array index
O(log n)LogarithmicBinary search, balanced-tree ops
O(n)LinearSingle loop, linear scan
O(n log n)LinearithmicMerge/heap/quick sort
O(n²)QuadraticNested loops, bubble sort
O(2ⁿ)ExponentialNaive recursion (subsets, Fibonacci)
O(n!)FactorialPermutations, brute-force TSP
📊
Time & space both matter. Analyze worst, average, and (sometimes) amortized cases — a dynamic array's Add is amortized O(1) despite occasional O(n) resizes. For recursion, watch the call-stack space too.
O(n²) — nested scan to find duplicates
for (int i = 0; i < n; i++) for (int j = i+1; j < n; j++) if (a[i] == a[j]) return true; // quadratic
O(n) — trade space for time with a HashSet
var seen = new HashSet<int>(); foreach (var x in a) if (!seen.Add(x)) return true; // O(1) lookups → O(n) total
⚖️
The space–time trade-off is everywhere. Hashing, memoization, and precomputed indexes spend memory to save time. Big-O hides constants — for small n, a "worse" algorithm with tiny constants can win. Always know your real input sizes.

#️⃣
Hash table internals
Hash → bucket. Collisions handled by chaining or open addressing; a good hash + low load factor keeps ops O(1) on average, O(n) worst case.
🌲
Balanced trees
Red-Black / AVL trees self-balance to guarantee O(log n). Back C#'s SortedDictionary & many DB indexes.
⛰️
Heap / Priority Queue
A complete binary tree where the root is min/max. The go-to for "K largest", Dijkstra, and scheduling.
🔤
Trie
A prefix tree of characters — autocomplete, spell-check, IP routing. Cost scales with key length, not dataset size.
Algorithms 03
Sorting & Searching
The most-studied algorithms — and a great lens on trade-offs (time vs space vs stability).
AlgorithmAverageWorstSpaceStable?
Quick SortO(n log n)O(n²)O(log n)No
Merge SortO(n log n)O(n log n)O(n)Yes
Heap SortO(n log n)O(n log n)O(1)No
Insertion SortO(n²)O(n²)O(1)Yes
Counting/RadixO(n + k)O(n·k)O(n + k)Yes
🔍
Binary search finds an element in a sorted array in O(log n) by halving the range each step. The technique generalizes far beyond arrays — "binary search on the answer" solves many optimization problems (find the smallest value that satisfies a monotonic condition).
Binary search — the canonical template
int lo = 0, hi = a.Length - 1; while (lo <= hi) { int mid = lo + (hi - lo) / 2; // avoids overflow if (a[mid] == target) return mid; if (a[mid] < target) lo = mid + 1; else hi = mid - 1; } return -1;
🧭
In practice, use the library sort. Array.Sort / List.Sort use tuned hybrid algorithms (introsort/Timsort). Know the theory to choose & reason; don't hand-roll sorts in production. Pick a stable sort when equal keys must keep their original order.
Algorithms 04
Trees & Graphs — Traversal & Paths
Graphs model networks, dependencies, maps, and social connections. Mastering traversal unlocks a huge class of problems.

BFS — Breadth-First Search

  • Explores level by level using a queue
  • Finds the shortest path in unweighted graphs
  • O(V + E) time
  • Use for: shortest hops, level order, nearest

DFS — Depth-First Search

  • Goes deep first via recursion / a stack
  • Detects cycles, topological sort, connectivity
  • O(V + E) time
  • Use for: paths, components, backtracking
🛣️
Dijkstra
Shortest path in a weighted graph (non-negative weights) using a min-heap. O(E log V).
⭐
A* Search
Dijkstra + a heuristic to guide the search toward the goal. The workhorse of pathfinding (games, maps).
📋
Topological Sort
Linear ordering of a DAG — task scheduling, build dependencies, course prerequisites.
🌐
Union-Find
Near-O(1) "are these connected?" with path compression. Powers Kruskal's MST & connectivity checks.
🗺️
Recognize the graph hiding in the problem. Maze, dependency chain, word ladder, friend network, state machine — all are graphs. Once you see nodes & edges, BFS/DFS/Dijkstra apply directly. Represent with an adjacency list for sparse graphs (most real ones).
Strategy 05
Algorithm Paradigms
The high-level strategies. Identifying the right paradigm is half the battle.
✂️
Divide & Conquer
Split into subproblems, solve, combine. Merge sort, quick sort, binary search, FFT.
💰
Greedy
Make the locally best choice at each step. Works when local optima → global optimum (Dijkstra, Huffman, interval scheduling).
🧠
Dynamic Programming
Overlapping subproblems + optimal substructure → memoize/tabulate. Knapsack, edit distance, LIS.
🔙
Backtracking
Try, recurse, undo. Explore the solution space, prune dead ends. N-Queens, sudoku, permutations.
🧩
Greedy vs DP: both build solutions incrementally, but greedy commits to a choice and never reconsiders, while DP explores all sub-solutions and picks the best. If a greedy choice can be "wrong" given later information, you need DP. Prove greedy works (exchange argument) before trusting it.
DP — memoized Fibonacci turns O(2ⁿ) into O(n)
var memo = new Dictionary<int, long>(); long Fib(int n) { if (n < 2) return n; if (memo.TryGetValue(n, out var v)) return v; // reuse return memo[n] = Fib(n-1) + Fib(n-2); }

Pattern recognition beats memorization. You can't memorize thousands of problems, but you can learn ~12 patterns and map any new problem to one. "Contiguous subarray with a constraint" → sliding window. "K most frequent" → heap. The signal in the prompt tells you the tool.
Method 07
How to Solve Any Problem
A repeatable framework — in interviews and real work alike. Communicate as you go.
1
Understand & clarify. Restate the problem, ask about constraints, edge cases, input size, and expected output. Confirm with an example.
2
Work an example by hand. Trace a small input to find the underlying structure and edge cases.
3
Brute force first. State the obvious solution & its complexity — it's a correct baseline and shows your thinking.
4
Optimize. Identify the bottleneck and the pattern that removes it (hashing, sorting, two pointers, DP). State the new complexity.
5
Code it cleanly. Readable names, handle edge cases, modular helpers. Talk through it as you write.
6
Test & verify. Walk through normal, edge, and boundary cases (empty, single, max, duplicates, negatives).
7
Analyze & discuss. Final time/space complexity, trade-offs, and how it scales or could be improved.
🗣️
Edge cases separate seniors from juniors: empty input, single element, duplicates, negatives, overflow, null, already-sorted, all-equal. Name them up front. And remember the real-world truth — clarity and correctness beat a clever one-liner nobody can maintain.
