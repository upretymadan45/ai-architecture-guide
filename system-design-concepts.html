Foundation 01
The Fundamentals — The Vocabulary of Scale
Every design decision is ultimately a conversation about these quantities. Get the definitions exact and the trade-offs become obvious.
⏱️
Latency
Time for a single operation end-to-end. Measure with percentiles (p50, p99), never just the average — tail latency is what users feel.
🚀
Throughput
Operations completed per unit time (RPS, QPS, MB/s). You can often trade latency for throughput (batching) and vice-versa.
📈
Scalability
The ability to handle growing load by adding resources. Linear scalability (2× resources → 2× capacity) is the goal; contention & coordination erode it.
🟢
Availability
Fraction of time the system is operational. Measured in "nines" — 99.99% ≈ 52 min/year downtime. Driven by redundancy & fast recovery.
🔒
Reliability
Probability the system performs correctly over time (no data loss/corruption). Available ≠ reliable — a system can be up but returning wrong answers.
🛠️
Maintainability
How easily the system is operated, understood, and evolved. Often the dominant long-term cost — favour simplicity & observability.
🔁
Consistency
Do all readers see the same data? Ranges from strong (always latest) to eventual (converges later). A core trade-off lever.
🧯
Fault Tolerance
The system keeps working when components fail. Achieved via redundancy, replication, retries, isolation, and graceful degradation.

📊
Availability "nines" cheat-sheet: 99% = 3.65 days/yr down · 99.9% = 8.76 hrs · 99.99% = 52.6 min · 99.999% = 5.26 min. Each extra nine is roughly 10× more expensive — match the target to the business need, don't over-buy.
Foundation 02
Characteristics of a Well-Designed System
"Good" isn't one property — it's a balanced set. A senior designs for the ones that matter for this product and consciously sacrifices the rest.

Signs of a healthy design

  • Scalable — grows by adding nodes, not rewrites
  • Available & resilient — survives node/zone failure
  • Performant — meets latency SLOs at p99
  • Loosely coupled — parts evolve independently
  • Observable — you can see & debug it in prod
  • Secure — least privilege, defence in depth
  • Simple — as few moving parts as the job allows
  • Cost-efficient — right-sized for the load
  • Evolvable — easy to change as needs change

Smells of a fragile design

  • Single points of failure (one DB, one node)
  • Tight coupling — one change ripples everywhere
  • Hidden synchronous chains that cascade failures
  • Unbounded queues/retries that amplify outages
  • No back-pressure → memory blow-ups under load
  • "It works on the happy path" — no failure design
  • No metrics/traces — debugging by guesswork
  • Premature complexity (microservices for a CRUD app)
  • Shared mutable state across services / a shared DB
⭐
The senior mindset: there is no "best" architecture — only the best fit for the constraints (load, budget, team, latency, consistency needs, time-to-market). Great design is the art of choosing which trade-offs to make and being able to defend them. Start simple; add complexity only when a real requirement forces it.

Method 03
The Approach — A Repeatable Design Process
In an interview or a real RFC, the process signals seniority more than any single answer. Drive it like this.
1
Clarify requirements & scope. Functional ("users post photos") and non-functional ("p99 < 200 ms, 99.99% uptime, 10M DAU"). Pin down read/write ratio, data size, and the few features that matter. Don't boil the ocean.
2
Estimate scale. Back-of-envelope: QPS, storage/yr, bandwidth, cache size. These numbers decide whether you need 1 box or 1,000.
3
Define the API & data model. The contract (REST/gRPC endpoints) and the core entities/access patterns. The access pattern drives the database choice — not the other way around.
4
Draw a high-level design. Client → LB → app → cache → DB, plus queues/CDN/search as needed. Get the boxes & arrows right before the details.
5
Deep-dive the hard parts. Pick the 1–2 components the scale stresses (the feed ranking, the ID generator, the hot partition) and detail them.
6
Identify bottlenecks & scale. Where does it break first? Add caching, replication, sharding, async, CDN — justify each with the numbers.
7
Address failures & trade-offs. What happens when a node/zone/region dies? Discuss consistency choices, single points of failure, and how you'd evolve it.
🎯
Always think out loud and state assumptions. "I'll assume 80/20 read/write and 500 bytes per record." Naming trade-offs ("I'm choosing eventual consistency here to keep writes fast; the feed can lag a few seconds") is exactly what distinguishes a senior answer.
Method 04
Back-of-the-Envelope Estimation
A few quick calculations turn hand-waving into engineering. Memorize the units and the method.

Handy constants

  • 1 day ≈ 86,400 s ≈ 105 s
  • 1 char ≈ 1 byte; 1 KB = 103, 1 MB = 106
  • 1 million writes/day ≈ ~12 writes/s average
  • Read-heavy apps: assume 100:1 to 1000:1 read/write
  • Peak ≈ 2–3× average; design for peak

What to estimate

  • QPS — average & peak reads/writes
  • Storage — bytes/record × records/day × years
  • Bandwidth — QPS × payload size
  • Memory — cache the hot 20% (80/20 rule)
  • Servers — total QPS ÷ per-node QPS
🧮
Worked example — a URL shortener at 100M new links/month:
• Writes: 100M ÷ (30 × 86,400) ≈ ~40 writes/s. With 100:1 read ratio → ~4,000 reads/s.
• Storage: 100M × 500 bytes ≈ 50 GB/month → ~3 TB over 5 years. Trivial for one sharded DB.
• Cache: hot 20% of reads → keep recent/popular mappings in Redis; ~a few GB serves most traffic.
Building Blocks 05
Scaling — Vertical vs Horizontal
There are only two ways to get more capacity: a bigger machine, or more machines. Each has a ceiling and a cost curve.

Vertical (scale up)

  • Bigger box — more CPU/RAM/disk
  • Simple: no app changes, no distribution
  • Hard ceiling (biggest machine available)
  • Single point of failure remains
  • Cost grows non-linearly at the top end

Horizontal (scale out)

  • More boxes behind a load balancer
  • Near-limitless ceiling; commodity hardware
  • Redundancy → fault tolerance built in
  • Requires statelessness, distribution, coordination
  • Adds complexity (consistency, balancing, ops)
🔑
Statelessness is the enabler. Horizontal scaling works because any node can serve any request. Push session/state to a shared store (Redis/DB) so app servers are interchangeable and can be added, removed, or replaced freely — the foundation of autoscaling and zero-downtime deploys.
📐
Amdahl & Universal Scalability: the serial fraction of your workload caps the speedup, and coordination/contention can make adding nodes slower past a point. Real-world scaling is sub-linear — minimize shared state, locks, and cross-node chatter to push the ceiling higher.
Building Blocks 06
Load Balancing & Traffic Distribution
A load balancer spreads requests across healthy nodes, removing single points of failure and enabling horizontal scale.
🔢
Round Robin
Rotate through servers in order. Simple & fair when nodes are equal and requests are uniform.
⚖️
Least Connections
Send to the node with the fewest active connections. Better for long-lived or uneven requests.
🧮
Hash / IP Hash
Map a key (client IP, session id) to a node consistently — useful for sticky routing without shared state.
🏋️
Weighted
Bigger nodes get proportionally more traffic. Good for heterogeneous fleets or gradual rollouts.
Layer 4 (Transport)Layer 7 (Application)
Operates onTCP/UDP, IP & portsHTTP — URLs, headers, cookies
SmartsFast, protocol-agnostic, no payload inspectionContent-based routing, TLS termination, path/host rules
Use forRaw throughput, any protocolAPIs, microservice routing, A/B, canaries
🩺
Health checks are the whole point. The LB continuously probes nodes and removes unhealthy ones from rotation, so failures are invisible to users. Pair with graceful draining (stop sending new traffic, let in-flight requests finish) for zero-downtime deploys. The LB itself must be redundant (active-active or active-passive) — don't replace one SPOF with another.
📌
Sticky sessions vs stateless: sticky (session affinity) pins a user to one node — simple but breaks rebalancing & fails when that node dies. Prefer stateless nodes + shared session store so any node can serve any user. Use stickiness only as a last resort.
Building Blocks 07
Caching — The Highest-Leverage Optimization
Caching trades a little staleness & memory for huge latency & load reductions. It exists at every layer of the stack.
🌍
CDN / Edge
Static assets & cacheable responses served from POPs near the user. Slashes latency & origin load. e.g. CloudFront, Cloudflare, Akamai.
🧠
In-memory (local)
Per-process cache (e.g. IMemoryCache). Fastest, but not shared and dies with the instance.
🗄️
Distributed (Redis)
Shared cache across all nodes. Survives restarts, enables scale-out, also does locks/pub-sub/rate-limiting.
🗃️
DB / query cache
Buffer pools, materialized views, and result caches inside or alongside the database.
How writes & reads interactCaching Strategies
StrategyHow it worksTrade-off
Cache-aside (lazy)App reads cache → on miss loads DB → populates cacheMost common & flexible; first request is slow; app owns invalidation
Read-throughCache library loads from DB on miss transparentlySimpler app code; couples cache to data source
Write-throughWrite to cache & DB synchronously togetherConsistent reads; slower writes
Write-behindWrite cache now, flush to DB asynchronouslyFast writes; risk of data loss if cache dies
Write-aroundWrite straight to DB, skip cacheAvoids cache churn for write-once data; cold reads
🌩️
The three failure modes:
• Stampede / dog-pile — a hot key expires, thousands hit the DB at once. Fix: single-flight lock, early/probabilistic refresh.
• Penetration — queries for non-existent keys bypass the cache. Fix: cache a null sentinel or a Bloom filter.
• Avalanche — many keys expire together. Fix: jitter the TTLs.
🧯
Invalidation is the hard part. Strategies: TTL expiry (simplest), write-time eviction, or versioned/tagged keys. Eviction policies when full: LRU (least recently used, the default), LFU (least frequently used), or TTL-based. Cache only data that's read-heavy, expensive to compute, and tolerant of slight staleness — and always track the hit ratio.
Building Blocks 08
Databases — Choosing the Right Store
The access pattern picks the database, not fashion. Match the data shape, query needs, and consistency requirements.
TypeBest forExamples
Relational (SQL)Structured data, joins, transactions, strong consistencyPostgreSQL, MySQL, SQL Server
Key-ValueSimple lookups, sessions, caches — ultra-fastRedis, DynamoDB, Memcached
DocumentSemi-structured/JSON, flexible schema, aggregatesMongoDB, Couchbase, Cosmos DB
Wide-ColumnHuge write volume, time-series, known query pathsCassandra, ScyllaDB, Bigtable, HBase
GraphHighly connected data, relationship traversalNeo4j, Neptune

Time-SeriesMetrics, IoT, append-heavy time-stamped dataInfluxDB, TimescaleDB, Prometheus
Blob / ObjectFiles, images, video, backupsS3, Azure Blob, GCS

SQL — pick when

  • You need ACID transactions & integrity
  • Complex queries, joins, ad-hoc reporting
  • The schema is well understood & stable
  • Strong consistency is required (money, inventory)

NoSQL — pick when

  • Massive scale / write throughput beyond one node
  • Flexible or rapidly-changing schema
  • Known, simple access patterns (key lookups)
  • Eventual consistency & availability are acceptable
⚖️
ACID vs BASE: SQL favours ACID (Atomic, Consistent, Isolated, Durable) — correctness first. Many NoSQL stores favour BASE (Basically Available, Soft-state, Eventually consistent) — availability & scale first. Neither is "better"; they sit on opposite ends of the consistency/availability trade-off. Polyglot persistence (different stores for different jobs) is normal.
🌲
Indexing & storage engines: indexes turn O(n) scans into O(log n) lookups but slow writes and cost space — index your query predicates, not everything. B-Tree engines (most SQL) are read-optimized; LSM-Tree engines (Cassandra, RocksDB) are write-optimized. Match the engine to your read/write ratio.
Building Blocks 09
Replication, Partitioning & Sharding
Replication copies data for availability & read scale; partitioning/sharding splits data for write scale & capacity. Most big systems do both.
Copy the dataReplication

Leader–Follower (primary–replica)

  • Writes go to the leader; reads fan out to followers
  • Scales reads; gives redundancy & failover
  • Replication lag → followers can serve stale reads
  • Failover promotes a follower if the leader dies

Multi-Leader / Leaderless

  • Multiple write nodes (multi-region, high availability)
  • Scales writes & survives region loss
  • Write conflicts → need resolution (LWW, vector clocks, CRDTs)
  • Leaderless (Dynamo-style) uses quorums: R + W > N
🔁
Sync vs async replication: synchronous replication guarantees no data loss but adds write latency and stalls if a replica is slow. Asynchronous is fast but can lose the last writes on failover. Many systems use semi-sync (ack from at least one replica) to balance durability and latency.
Split the dataPartitioning / Sharding
StrategyHowWatch out for
RangePartition by key ranges (A–M, N–Z, by date)Hot spots if data/traffic is skewed (e.g. "today")
HashPartition by hash(key) % NEven spread, but range queries scatter; resharding moves everything
Consistent hashingKeys & nodes on a ring; add/remove moves minimal dataThe standard for elastic clusters (Cassandra, Dynamo)
Directory / lookupA lookup service maps key → shardFlexible, but the directory is a new SPOF/bottleneck
🔥
The hot-partition (celebrity) problem: a single key (a viral post, a huge customer) overwhelms one shard. Mitigate by choosing a high-cardinality shard key, adding a salt/suffix to spread a hot key, or caching it separately. Choosing the shard key is the most consequential decision — a bad key means painful resharding later.

Theory 10
CAP, PACELC & Consistency Models
The theorems that govern every distributed-data decision. Memorize them; they'll frame half your design discussions.
📜
CAP theorem: in the presence of a network Partition you can guarantee Consistency or Availability, not both. Since partitions are inevitable, the real choice is CP (refuse/limit requests to stay consistent) vs AP (stay available, reconcile later).
ChoiceOn partitionFits
CPSacrifice availability to avoid stale/conflicting dataBanking, inventory, bookings, leader election
APKeep serving, accept temporary inconsistencyFeeds, carts, catalogs, DNS, recommendations
➕
PACELC refines CAP: if Partition → choose A or C; Else (normal operation) → choose Latency or Consistency. Even with no failures, strong consistency costs coordination round-trips. This is why "eventually consistent" systems feel faster.
A spectrum, not a switchConsistency Models
🔐
Strong / Linearizable
Every read sees the latest write, as if there were one copy. Simplest to reason about, most expensive. e.g. a single-leader RDBMS, Spanner.
⏳
Eventual
Replicas converge "eventually" if writes stop. Cheapest & most available. e.g. DNS, Dynamo, S3 (historically).
👤
Read-your-writes
A user always sees their own latest write (even if others lag). Crucial for "I posted but don't see it" UX.
🔗
Causal / Monotonic
Preserves cause→effect ordering and never shows data going "backwards." A practical middle ground.
🧩
Apply consistency per use-case, not globally. A bank balance needs strong consistency; a "likes" counter is fine eventually consistent. Mixing models within one product is normal and smart — pay for consistency only where correctness demands it.
Building Blocks 11
Messaging, Queues & Asynchronous Processing
Async decoupling is how systems absorb spikes, stay resilient, and scale producers and consumers independently.
📨
Message Queue
Point-to-point: one message → one consumer in a competing group. Load-levels spikes & distributes work. e.g. SQS, RabbitMQ, Azure Service Bus.
📢
Pub/Sub (Topic)
One event → many subscribers, each independent. Enables fan-out & loose coupling. e.g. SNS, Service Bus topics.
📜
Event Streaming Log
Append-only, replayable, retained log. Consumers track offsets & can re-read. e.g. Kafka, Kinesis, Event Hubs.
☠️
Dead-Letter Queue
Where poison messages land after max retries, so one bad message can't block the pipeline. Always pair with a replay plan.

Why async wins at scale

  • Decouples producer & consumer in time & speed
  • Absorbs traffic spikes (buffer, then drain)
  • Resilient — broker holds messages if a consumer is down
  • Independent scaling & deployment of each side
  • Natural fan-out to many downstream consumers

Costs you must handle

  • Eventual consistency & harder reasoning
  • Duplicate delivery → need idempotent consumers
  • Ordering only within a partition/key
  • Visibility/debugging across hops needs tracing
  • Back-pressure & DLQ handling required
⚠️
The dual-write problem & the Outbox pattern: writing to your DB and publishing to a broker as two steps can fail between them → inconsistency. Fix with the Transactional Outbox: write the event to an outbox table in the same DB transaction, then a relay reliably publishes it. Make consumers idempotent (dedupe by message id) because delivery is usually at-least-once.
🔀
Saga pattern for distributed workflows: instead of a distributed transaction, chain local transactions, each emitting an event; on failure, run compensating actions to undo prior steps. Choreography (services react to events) is loosely coupled; orchestration (a central coordinator) is more visible & testable.

How clients and services talk. Pick the protocol that matches the interaction shape (request/response, streaming, real-time).
StyleStrengthsUse when

gRPCBinary (protobuf), fast, streaming, strong contractsInternal service-to-service, low-latency, polyglot

🚪
API Gateway: a single entry point that handles routing, authentication, rate limiting, TLS termination, request aggregation, and versioning — so individual services don't each reimplement it. A BFF (Backend-for-Frontend) tailors responses per client type (web vs mobile).
🔢
Idempotency & versioning: design write APIs to be safely retryable (idempotency keys) so a client retry after a timeout doesn't double-charge. Version your APIs (/v1, headers) and evolve additively to avoid breaking existing clients.
Architecture 13
Architectural Patterns & Styles
The shapes systems take. Each optimizes for different things — there is no universally correct choice.

Monolith

  • One deployable; simple to build, test, deploy early
  • Fast in-process calls, easy transactions
  • Scales as a unit; one bug can take all down
  • Gets hard to evolve as the codebase & team grow

Modular Monolith

  • One deployable, but clear internal module boundaries
  • Most of the monolith's simplicity
  • Enforced seams ease a later split to services
  • The recommended starting point for most

Microservices

  • Independently deployable services per capability
  • Independent scaling, tech choice, team ownership
  • Each owns its data (no shared DB)
  • Heavy ops, network, & consistency complexity

Event-Driven / SOA / Serverless

  • Event-driven: components react to events, loosely coupled
  • SOA: coarse-grained shared services + ESB
  • Serverless (FaaS): functions, auto-scale, pay-per-use, cold starts
  • Cell-based: isolated cells limit blast radius
🧱
Supporting tactical patterns: CQRS (separate read/write models), Event Sourcing (store changes as events), Strangler Fig (incrementally replace a legacy system), Sidecar / Service Mesh (offload cross-cutting concerns like mTLS, retries to a proxy), and the Saga & Outbox patterns for distributed consistency.
⚖️
Conway's Law: your architecture will mirror your org's communication structure. Microservices need autonomous teams to pay off — without that, you get a distributed monolith. Start with a modular monolith and extract services only when a real driver (scale, team autonomy, release cadence) appears.
Operations 14
Reliability, Resilience & Fault Tolerance
Distributed systems fail partially and constantly. Design for failure as the default state, not the exception.
🔁
Retry + Backoff
Retry transient failures with exponential backoff + jitter. Only retry idempotent ops; cap attempts to avoid amplifying an outage.
⚡
Circuit Breaker
After N failures, "open" and fail fast for a cooldown so you stop hammering a sick dependency; half-open to test recovery.
🧱
Bulkhead
Isolate resource pools so one slow/failing dependency can't starve the whole app (like watertight ship compartments).
⏱️
Timeout
Bound every external call. A request waiting forever ties up threads/connections and cascades into a full outage.
📉
Graceful Degradation
Shed non-essential features under stress (serve cached/stale data, hide recommendations) instead of failing entirely.
🚦
Rate Limiting / Throttling
Protect the system & ensure fairness. Token bucket, leaky bucket, sliding window. Reject early with 429.
🛟
Redundancy & Failover
No single points of failure — N+1 nodes, multi-AZ/region, automatic failover with health checks.
🐒
Chaos Engineering
Deliberately inject failures (kill nodes, add latency) in prod-like envs to prove resilience before reality does.
Recovery objectivesDisaster Recovery — RTO & RPO
MetricQuestion it answersLower value means
RTO (Recovery Time Objective)How fast must we be back up?More redundancy/automation (hot standby)
RPO (Recovery Point Objective)How much data can we afford to lose?More frequent replication/backups
🗺️
DR strategies by cost/speed: Backup & Restore (cheap, slow) → Pilot Light → Warm Standby → Multi-Region Active-Active (expensive, near-zero RTO/RPO). Choose by how much an hour of downtime/data-loss actually costs the business.
🎯
SLA / SLO / SLI: an SLI is the measured signal (e.g. p99 latency), an SLO is the internal target (p99 < 200 ms 99.9% of the time), and an SLA is the external promise with penalties. Set SLOs first; use an error budget to balance reliability work against shipping features.
Operations 15
Observability — You Can't Operate What You Can't See
Monitoring tells you when something is wrong; observability lets you ask why without shipping new code.
📊
Metrics
Numeric time-series (RPS, latency percentiles, error rate, CPU). Cheap to store, great for dashboards & alerts. e.g. Prometheus, Grafana.
📝
Logs
Discrete, structured event records. Invaluable for detail; expensive at volume. Centralize & index. e.g. ELK, Loki.
🧵
Traces
Follow one request across services via a correlation/trace id. Pinpoints which hop is slow/broken. e.g. OpenTelemetry, Jaeger.
🔔
Alerting
Alert on symptoms users feel (SLO breaches), not every cause. Avoid alert fatigue — actionable, owned, runbooked.
🚦
The Four Golden Signals (Google SRE): Latency, Traffic, Errors, and Saturation. If you monitor only four things, monitor these. Track latency as percentiles (p50/p95/p99), never just the mean — the average hides the pain.
🧰
Health checks & correlation IDs: expose liveness (am I running?) and readiness (can I serve traffic?) endpoints for the LB/orchestrator. Propagate a correlation ID through every hop so a single user request can be reconstructed end-to-end across logs, metrics, and traces.
Operations 16
Security & Networking Foundations
Security is a cross-cutting design concern, not a final checklist. Build defence in depth and least privilege in from the start.
🔑
AuthN vs AuthZ
Authentication = who you are (OAuth2/OIDC, JWT, sessions). Authorization = what you may do (RBAC, ABAC, policies). Always enforce on the server.
🔒
Encryption
In transit (TLS everywhere, mTLS between services) and at rest (KMS-managed keys). Hash passwords (bcrypt/argon2), never store plaintext.
🛡️
Network isolation
VPC/VNet, private subnets for data tiers, security groups (stateful), NACLs (stateless), bastion hosts. Expose only what must be public.
🧱
Edge protection
WAF, DDoS protection, rate limiting, bot mitigation at the edge/CDN before traffic reaches your origin.
🔐
Core principles: least privilege (grant the minimum, for code and people), defence in depth (multiple independent layers), zero trust (verify every request, even internal), validate & sanitize all input at boundaries (OWASP Top 10), keep secrets in a vault, and patch dependencies. Security failures are usually design failures.
🌐
Networking essentials: DNS resolves names → IPs (and enables geo/failover routing); CDN caches content at the edge; reverse proxy (Nginx/Envoy) terminates TLS & routes; API gateway centralizes cross-cutting concerns. Latency is dominated by round-trips — keep data and compute close to users.
Reference 17
The Trade-off Cheat-Sheet
Every system-design decision is a trade-off. This is the senior's mental index of the big ones.
You want more…You usually give up…The lever
ConsistencyAvailability / LatencyCAP / PACELC — sync replication, quorums
AvailabilityConsistency (accept staleness)Async replication, eventual consistency
Lower latencyFreshness / costCaching, CDN, denormalization, read replicas
Higher throughputPer-request latencyBatching, async queues, buffering
ScalabilitySimplicityHorizontal scaling, sharding, statelessness
Fast readsWrite performance / storageIndexes, materialized views, denormalization
Fast writesRead performanceLSM engines, write-behind cache, normalization
Flexibility (schema)Query power / integrityNoSQL document vs relational SQL
Strong durabilityWrite latencySync replication, fsync, multi-AZ writes
Loose couplingSimplicity / consistencyAsync messaging, eventual consistency
Team autonomy / independent deployOperational complexityMicroservices vs monolith
Cost efficiencyPerformance headroomRight-sizing, autoscaling, spot/serverless
🧠
How to use it: name the requirement, find the lever, state what you're sacrificing and why it's acceptable here. "This is read-heavy and tolerates 2-second staleness, so I'll add read replicas + a cache and accept eventual consistency to hit the latency SLO." That sentence is senior system design.
📏
Guiding maxims: KISS (keep it simple), YAGNI (don't build for imagined needs), "premature optimization is the root of all evil", and "make it work, make it right, make it fast — in that order." Complexity is a cost you pay forever; add it only when the requirement demands it.
Part B — Applied
Worked System Design Examples
Concepts become real when applied. Here are condensed, senior-level walkthroughs of classic designs — requirements, key decisions, and the trade-offs.
Example 1TinyURL — a URL Shortener

Requirements: shorten a long URL to a unique short code; redirect on access; very read-heavy (~100:1); low latency; high availability. Analytics is a bonus.

Key decisions

  • Generate a unique short code: base-62 encode an auto-increment/Snowflake ID, or hash + collision check
  • Store shortCode → longURL in a key-value/SQL store; shard by code
  • Cache hot mappings in Redis (read-heavy → huge hit ratio)
  • Redirect with 301/302; put a CDN/edge in front
  • Counters & analytics via async events (don't block redirects)

Trade-offs

  • Eventual consistency for analytics is fine
  • 302 (no caching) for accurate clicks vs 301 (cacheable) for speed
  • Pre-generate ID ranges per node to avoid a write bottleneck
  • Read replicas + cache absorb the 100:1 read skew

Example 2News Feed (Twitter/Instagram timeline)

Requirements: users follow others; show a personalized, recent feed fast; massive read scale; some followers have millions of fans (celebrities).

Fan-out on write (push)

  • On post, push the post id into every follower's feed cache
  • Reads are O(1) & instant — great for most users
  • Expensive for celebrities (millions of writes per post)

Fan-out on read (pull)

  • Build the feed at read time by querying followees
  • Cheap writes; expensive, slow reads at scale
  • Better for users who follow many / celebrities
🔀
The real answer is hybrid: fan-out-on-write for normal users (fast reads), but fan-out-on-read for celebrity accounts (avoid the write storm) — merge the two at read time. Cache feeds in Redis, store posts in a wide-column store, precompute ranking, and accept eventual consistency (a few seconds of lag is fine for a feed).
Example 3Real-Time Chat (WhatsApp/Slack)

Requirements: 1:1 & group messaging, real-time delivery, online presence, delivery/read receipts, message history, ordering.

1
Transport: persistent WebSocket connections via a gateway layer for bidirectional, low-latency push.
2
Routing: a message goes gateway → message service → recipient's connected gateway (via a pub/sub bus like Redis/Kafka so any gateway can reach any user).
3
Storage: persist messages in a write-optimized wide-column store (Cassandra) partitioned by conversation; order by a time-based id.
4
Offline: if the recipient is offline, queue the message and push a notification; deliver on reconnect.
5
Presence: track online/last-seen in Redis with heartbeats & TTLs.
📩
Ordering & delivery: guarantee order within a conversation using monotonic message ids (Snowflake). Use acks + client retries for at-least-once delivery, and dedupe by message id. Group chat = fan-out the message to all members' gateways.
Example 4Distributed Rate Limiter

Requirements: limit each client to N requests per window across a fleet of servers; low latency; fail open or closed by policy.

AlgorithmBehaviourNote
Token bucketTokens refill at a steady rate; each request spends oneAllows bursts up to bucket size; most popular
Leaky bucketRequests drain at a fixed rate; overflow rejectedSmooths bursts into a steady stream
Fixed windowCount per fixed intervalSimple but allows 2× burst at window edges
Sliding windowRolling count over the last intervalSmoother & fairer; slightly more state
⚡
Distributed implementation: keep counters in Redis (atomic INCR + TTL, or a Lua script for token bucket) so all nodes share state. Reject over-limit with HTTP 429 + Retry-After. Decide fail-open (allow if Redis is down — availability) vs fail-closed (block — protection) per the use-case.
Example 5Video Streaming & Large File Storage (YouTube/Netflix)
1
Upload & store: raw files in object storage (S3); metadata (title, owner, URLs) in a database.
2
Transcode asynchronously into multiple resolutions/bitrates via a queue + worker fleet (adaptive bitrate streaming, HLS/DASH).
3
Deliver via CDN — segments cached at the edge near users; the player adapts quality to bandwidth.
4
Scale reads with the CDN (origin barely touched); use presigned URLs for secure, direct-to-storage access.
🎬
Pattern reuse: notice how the same building blocks recombine — object storage + metadata DB, async queues for heavy work, CDN for read scale, and adaptive streaming for variable networks. Most "new" designs are just the fundamentals composed for a new problem.
✅
The meta-lesson: almost every large-scale system reduces to the same toolkit — load balancers, stateless app tiers, caches, the right database with replication & sharding, async queues, a CDN, and thoughtful consistency choices. Master the building blocks and the trade-offs, and you can design anything by composing them to fit the requirements.
