Topic 01
async / await, Task, Threading & Cancellation
Asynchrony is about not blocking a thread while waiting (mostly I/O). Parallelism is about using more threads to do more CPU work. They are different problems — don't confuse them.
⏳
async / await
Syntactic sugar over a state machine. await releases the current thread back to the pool until the awaited operation completes, then resumes. Ideal for I/O (DB, HTTP, files).
📦
Task / Task<T>
A promise of future work/result. It is not a thread — it's a unit of work that may run on a pool thread, inline, or just represent an I/O completion.
🧵
Thread
An OS-level execution unit (~1MB stack). Expensive to create. You rarely create raw threads today — the thread pool + Task manage them for you.
🛑
CancellationToken
Cooperative cancellation. A CancellationTokenSource signals; the token is passed down and checked via ThrowIfCancellationRequested() or honored by async APIs.
💡
Mental model: Threads are waiters in a restaurant; Tasks are orders. With async/await, a waiter who places an order in the kitchen (I/O) doesn't stand and wait — they serve other tables and come back when the food (result) is ready. That's why a few threads can serve thousands of concurrent requests.
Cancellation done right — flow the token all the way down
public async Task<IActionResult> GetReport(CancellationToken ct) { // ASP.NET Core injects ct tied to the client connection. // If the user closes the tab, ct is cancelled and work stops. var data = await _repo.GetSalesAsync(ct); // passed to EF Core return Ok(data); } // Manual cancellation source (e.g. timeout): using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5)); await httpClient.GetAsync(url, cts.Token);
⚠️
Common pitfalls: blocking on async with .Result / .Wait() (deadlocks & thread-pool starvation), forgetting to pass CancellationToken downstream, and using Task.Run to "make code async" on the server — it just steals a pool thread, it doesn't add concurrency.

IEnumerable vs IQueryable vs IAsyncEnumerable
The single most important distinction: where does the work execute, and when is it materialized?
AspectIEnumerable<T>IQueryable<T>IAsyncEnumerable<T>
Executes where In-memory (client / your app) Translated to the data source (SQL on the DB server) In-memory or DB, streamed asynchronously
Filtering LINQ-to-Objects — pulls all rows, filters in RAM LINQ-to-Entities — builds an expression tree → WHERE in SQL Streams items one-by-one with await foreach
Best for Collections already in memory Database queries you want pushed server-side Large/continuous result sets without buffering all in memory
Risk Over-fetching if used too early on a DB query Calling unsupported methods → translation error Holding the DB connection open while you stream
DANGER — switches to in-memory too early
_context.Orders .AsEnumerable() // ← pulls EVERY order into memory NOW .Where(o => o.CustomerId == id) // filtered in RAM, not in SQL .ToList();
CORRECT — stays as IQueryable, filters in SQL
_context.Orders .Where(o => o.CustomerId == id) // → WHERE CustomerId = @id in SQL .ToListAsync(); // only matching rows leave the DB
IAsyncEnumerable — stream without buffering everything
await foreach (var order in _context.Orders.AsAsyncEnumerable()) { Process(order); // each row processed as it arrives }
Topic 03
Dependency Injection Lifetimes
Pick the wrong lifetime and you get either captive dependencies, shared mutable state, or wasted allocations.
🔂
Transient
A new instance every time it is requested. Use for lightweight, stateless services. Safe but can allocate a lot.
🔄
Scoped
One instance per request/scope (per HTTP request in ASP.NET Core). The default for DbContext and most app services.
♾️
Singleton
One instance for the whole app lifetime. Great for config, caches, HttpClientFactory. Must be thread-safe.
🚫
The captive dependency trap: injecting a Scoped (or Transient) service into a Singleton captures it forever — the scoped service never gets disposed/refreshed, and a per-request DbContext ends up shared across all requests (not thread-safe → data corruption, ObjectDisposedException). See the full fix in the Q&A section.

Transient✅ OK✅ OK✅ OK

Topic 04
EF Core — Tracking, Include, Lazy Loading & the N+1 Problem
Most "the app is slow" tickets in a .NET shop trace back to one of these four things.

Tracking (default)

  • EF keeps a snapshot of every entity in the change tracker
  • Required when you intend to Update / SaveChanges
  • Costs memory + CPU for the snapshot

AsNoTracking()

  • No snapshot — read-only, faster, less memory
  • Perfect for GET / listing / reporting queries
  • Changes won't be persisted (by design)
Read-only listing → AsNoTracking
var customers = await _context.Customers .AsNoTracking() .ToListAsync();
The classic killerThe N+1 Query Problem

You run 1 query to load a list, then N more queries (one per row) to load each related entity. 1 + N round-trips to the database = a page that takes seconds.

N+1 — 1 query for customers, then 1 per customer for orders
var customers = _context.Customers.ToList(); // 1 query foreach (var c in customers) var orders = c.Orders.ToList(); // +1 query EACH → N queries
FIXED — eager-load with Include = 1 (or 2) queries total
var customers = await _context.Customers .Include(c => c.Orders) .ThenInclude(o => o.LineItems) // load the whole graph in one go .AsNoTracking() .ToListAsync();

Lazy Loading

  • Related data loaded automatically on first property access
  • Convenient — but silently triggers N+1
  • Needs proxies + virtual navigation properties

Eager Loading (Include)

  • You declare what to load up front
  • Predictable query count, no hidden round-trips
  • Preferred for web request paths

Topic 05
Cloud Networking — VNet / VPC, Subnets & Security Groups
The minimum networking vocabulary every backend engineer should own (terms shown for AWS & Azure).
🌐
VPC / VNet
Your private, isolated network in the cloud with its own IP address range (CIDR, e.g. 10.0.0.0/16). Nothing gets in or out unless you allow it.
🧩
Subnet
A slice of the VPC's IP range. Public subnets reach the internet (via an Internet Gateway); private subnets (DBs, app servers) do not.
🛡️
Security Group
Stateful virtual firewall on the resource (VM/instance). Allow-rules only; return traffic is automatically permitted. E.g. allow 443 from anywhere, 5432 only from the app subnet.
📋
NACL (Network ACL)
Stateless firewall at the subnet boundary. Supports allow and deny rules; you must define inbound and outbound separately. A coarse second layer.
🔐
Defence in depth, least privilege: put databases in private subnets, expose only a load balancer publicly, lock Security Groups to specific ports/sources (never 0.0.0.0/0 on a DB), use NACLs as a coarse subnet-level backstop, and reference security groups by ID (SG-to-SG rules) instead of hard-coding IPs.

Topic 07
C# Type System — Value vs Reference, Boxing, Records & Strings
Where data lives (stack vs heap), how it's copied, and the hidden allocations that surprise people in hot paths.

Value types (struct, enum, primitives)

  • Hold the data directly; copied by value on assignment/pass
  • Usually stack-allocated (or inline inside a heap object)
  • No null by default (unless Nullable<T> / T?)
  • Great for small, immutable, short-lived data

Reference types (class, interface, delegate, string)

  • Variable holds a reference; object lives on the heap
  • Copied by reference — two variables, one object
  • Can be null; managed by the GC
  • Equality is identity by default (unless overridden)
📦
Boxing & unboxing: assigning a value type to object / an interface boxes it — a heap allocation + copy. Unboxing copies back and casts. In tight loops this is a silent performance killer. Use generics (List<int>, not ArrayList) and avoid object APIs on value types.
Hidden boxing
int n = 42; object boxed = n; // boxing — heap alloc + copy int back = (int)boxed; // unboxing — copy back ArrayList list = new(); list.Add(n); // boxes every int
No boxing — generics keep value types unboxed
List<int> list = new(); list.Add(42); // stays an int, no allocation
🧊
record / record struct
Immutable-by-convention types with value-based equality, with-expressions, and deconstruction. Ideal for DTOs & domain values. record class is a reference type; record struct is a value type.
🔤
string immutability
Strings are immutable reference types — every "mutation" makes a new object. Concatenating in a loop allocates repeatedly; use StringBuilder for many edits.
🎯
Span<T> / Memory<T>
Allocation-free views over arrays/strings/stack memory. Span<T> is a ref struct (stack-only); enables slicing & parsing with zero copies.
🪢
ref / in / out
ref passes by reference (read/write), in passes a readonly reference (avoids copying big structs), out must be assigned by the callee.
💡
struct vs class heuristic: reach for a struct only when the type is small (≈16 bytes), immutable, logically a single value, and short-lived. Otherwise use a class — large mutable structs cause expensive copies and subtle bugs.
Topic 08
Memory Management — Garbage Collection, IDisposable & Leaks
.NET is managed, but "managed" ≠ "leak-proof". Senior engineers know what the GC does and where it can't help.
👶
Gen 0 / 1 / 2
Generational GC: new objects start in Gen 0 (collected often & cheaply). Survivors are promoted to Gen 1, then Gen 2 (collected rarely). Most objects die young — that's the design assumption.
🐘
Large Object Heap
Objects ≥ 85 KB go on the LOH, collected with Gen 2 and not compacted by default → fragmentation. Pool large buffers (ArrayPool<T>) instead of reallocating.
🧹
IDisposable / using
Deterministic cleanup of unmanaged resources (sockets, file handles, DB connections). using guarantees Dispose() even on exceptions.
⚰️
Finalizers
A safety net (~Type()) for unmanaged resources if Dispose wasn't called. They delay collection (2 GC cycles) — implement the full Dispose pattern, don't rely on finalizers.
🩸
Yes, .NET can leak. Common causes: lingering event handler subscriptions (subscriber kept alive by publisher), static collections that only grow, undisposed IDisposables, captured variables in long-lived closures/timers, and captive dependencies in DI. The GC can't collect what's still reachable.
Full Dispose pattern
public class FileService : IDisposable { private FileStream? _stream; private bool _disposed; public void Dispose() { Dispose(true); GC.SuppressFinalize(this); // no need to finalize, already cleaned } protected virtual void Dispose(bool disposing) { if (_disposed) return; if (disposing) _stream?.Dispose(); // managed resources _disposed = true; } }
⚡
Reduce GC pressure: avoid needless allocations in hot paths (boxing, LINQ closures, string concatenation), use Span<T>/stackalloc, pool buffers with ArrayPool<T>, prefer ValueTask for hot async that often completes synchronously, and stream large payloads instead of buffering. Measure with the GC counters / dotnet-counters.
Topic 09
Collections & LINQ — Pick the Right Structure, Know Deferred Execution
Choosing the wrong collection turns an O(1) lookup into an O(n) scan; misunderstanding deferred execution causes double-enumeration and surprise queries.
CollectionLookupUse when
List<T>O(n) by value, O(1) by indexOrdered, index access, append-heavy
Dictionary<K,V>O(1) average by keyKeyed lookups, caches, de-dup by key
HashSet<T>O(1) average containsUniqueness, fast membership tests
Queue<T> / Stack<T>O(1) enqueue/pushFIFO / LIFO processing
ConcurrentDictionaryO(1) thread-safeShared mutable map across threads
ImmutableArray / ListO(1)/O(log n)Shared read-only data, snapshot semantics
⏳
Deferred (lazy) execution: LINQ operators like Where, Select, OrderBy don't run until you enumerate (foreach, ToList, Count, First). The query is re-executed every time you iterate — so enumerating twice hits the DB twice. Materialize once with ToList() when you'll reuse the result.
Double enumeration — query runs twice
var q = _context.Orders.Where(o => o.Active); // nothing executed yet var count = q.Count(); // SQL round-trip #1 foreach (var o in q) { ... } // SQL round-trip #2 — re-queried!
Materialize once, reuse
var orders = await _context.Orders .Where(o => o.Active) .ToListAsync(); // one round-trip var count = orders.Count; // in-memory, free
🎯
Custom key equality: using a type as a Dictionary/HashSet key requires correct GetHashCode + Equals (records give you this for free). A mutable key whose hash changes after insertion becomes unfindable — a classic subtle bug.
Topic 10
Concurrency & Thread Safety — Locks, Atomics & Deadlocks
Async solves "don't block on I/O". This solves "multiple threads touching shared mutable state at once."
🔒
lock / Monitor
Mutual exclusion over a critical section. Lock on a private readonly object, keep the section tiny, and never await inside a lock.
➕
Interlocked
Lock-free atomic operations (Increment, Add, CompareExchange) for simple counters/flags — far cheaper than a lock.
🚦
SemaphoreSlim
Async-friendly throttle (await WaitAsync()). The correct way to limit concurrency or guard a critical section in async code.
🧊
Immutable / Concurrent
Best of all: avoid shared mutable state. Use immutable types or System.Collections.Concurrent so no explicit locking is needed.
💀
Deadlocks happen when two threads each hold a lock the other needs (circular wait). Avoid by acquiring locks in a consistent global order, keeping critical sections short, using timeouts, and preferring lock-free or message-passing designs. Classic .NET deadlock: blocking on async with .Result/.Wait() on a thread with a SynchronizationContext.
Race condition — lost updates
private int _count; public void Add() => _count++; // read-modify-write is NOT atomic
Atomic — Interlocked
private int _count; public void Add() => Interlocked.Increment(ref _count);
Throttle concurrency in async with SemaphoreSlim
private readonly SemaphoreSlim _gate = new(5); // max 5 at once public async Task ProcessAsync(Item item) { await _gate.WaitAsync(); try { await DoWorkAsync(item); } finally { _gate.Release(); } // always release }
✅
Volatile vs lock: volatile only guarantees visibility/ordering of a single field — it does not make compound operations atomic. For anything beyond a single flag read, use Interlocked or a lock.

ASP.NET Core — Pipeline, Middleware, Filters & Config
A request flows through an ordered middleware pipeline. Knowing the order and the extension points is core senior knowledge.
Request pipeline (order matters)
flowchart LR
  R(("Request")) --> EX["Exception\nHandler"]
  EX --> HSTS["HSTS / HTTPS\nRedirect"]
  HSTS --> ST["Static Files"]
  ST --> RT["Routing"]

  AUTHN --> AUTHZ["Authorization"]

🧱
Middleware
Cross-cutting components in a pipeline; each can short-circuit or call next(). Order is critical (auth before authorization, exception handler first).
🪝
Filters
MVC-scoped hooks: Authorization → Resource → Action → Exception → Result. Use for model validation, logging, caching around actions.
⚙️
Options pattern
Strongly-typed config via IOptions<T> / IOptionsSnapshot<T> (per-request) / IOptionsMonitor<T> (live reload).
🔑
AuthN vs AuthZ
Authentication = who are you (JWT/cookie). Authorization = what may you do (roles, policies, claims). They run as separate middleware in that order.
Custom middleware
public class TimingMiddleware { private readonly RequestDelegate _next; public TimingMiddleware(RequestDelegate next) => _next = next; public async Task InvokeAsync(HttpContext ctx) { var sw = Stopwatch.StartNew(); await _next(ctx); // call the rest of the pipeline sw.Stop(); ctx.Response.Headers["X-Elapsed-ms"] = sw.ElapsedMilliseconds.ToString(); } }
🧩
Middleware vs Filter: middleware is global and runs for every request (even static files); filters are MVC-aware and run only around actions, with access to model state and action context. Use middleware for truly cross-cutting infra (logging, auth, exception handling); filters for action-level concerns (validation, result shaping).
Topic 13
HttpClient, Resilience & Microservices Patterns
Distributed systems fail partially and constantly. Seniors design for failure rather than assuming the happy path.
🔌
Socket exhaustion: new HttpClient() per request leaks sockets stuck in TIME_WAIT and eventually fails. Use IHttpClientFactory (typed/named clients) so handlers are pooled and DNS rotation works. Never wrap a single HttpClient in using.
🔁
Retry + backoff
Retry transient failures with exponential backoff + jitter. Only retry idempotent operations; cap attempts to avoid amplifying an outage.
⚡
Circuit breaker
After N failures, "open" the circuit and fail fast for a cooldown — stop hammering a sick dependency, then half-open to test recovery. Polly provides both.
⏱️
Timeout + bulkhead
Bound every external call with a timeout; isolate resource pools (bulkhead) so one slow dependency can't starve the whole app.
📤
Outbox / idempotency
For cross-service consistency, write events to an outbox in the same transaction and publish reliably; make consumers idempotent (dedupe by message id).
Typed client with Polly resilience (retry + circuit breaker)
builder.Services.AddHttpClient<OrdersClient>(c => c.BaseAddress = new("https://orders.api")) .AddStandardResilienceHandler(); // retry + breaker + timeout (Microsoft.Extensions.Http.Resilience)
🧠
Consistency trade-offs: distributed transactions are usually the wrong tool. Prefer eventual consistency via async messaging + the outbox + saga patterns, design for idempotency and at-least-once delivery, and make operations safe to retry.

Topic 15
Testing — Unit, Integration & the Right Boundaries
Seniors are expected to make code testable by design and to know what to mock vs what to test for real.

Unit tests

  • Test one class/method in isolation; mock its dependencies (Moq/NSubstitute)
  • Fast, deterministic, run on every build
  • Arrange → Act → Assert; one logical assertion per test

Integration tests

  • Test components together (DB, pipeline) via WebApplicationFactory
  • Use Testcontainers / a real DB over in-memory for EF fidelity
  • Slower but catch wiring, SQL translation & config bugs
🎭
Mock at boundaries, not internals. Mock external dependencies you don't own (HTTP APIs, time, message buses) — inject IClock/TimeProvider instead of DateTime.Now. Don't mock the thing under test or value objects. The EF Core in-memory provider isn't a real database — it won't catch SQL/translation issues; prefer a containerized real DB for repository tests.
Testable design — inject TimeProvider, mock the dependency
public class TokenService { private readonly TimeProvider _clock; public TokenService(TimeProvider clock) => _clock = clock; public bool IsExpired(Token t) => t.ExpiresAt < _clock.GetUtcNow(); } // test — control time, no flakiness var clock = new FakeTimeProvider(DateTimeOffset.Parse("2026-01-01Z")); var svc = new TokenService(clock); Assert.True(svc.IsExpired(expiredToken));
🔺
Test pyramid: many fast unit tests, fewer integration tests, very few end-to-end. Aim for meaningful coverage of business logic and edge cases — not a 100% number. Tests are a design tool: hard-to-test code is usually badly-coupled code.
Topic 16
Microservices & Distributed Systems
Microservices trade in-process simplicity for independent deployability — and inherit the hard problems of distributed systems. A senior knows the cost before reaching for them.
🧭
Start with a modular monolith. Split into services only when you have a real driver: independent scaling, team autonomy, or differing release cadences. Premature microservices give you a distributed monolith — all the network pain, none of the independence.
📦
Service boundaries
Draw boundaries around business capabilities (DDD bounded contexts), not technical layers. Each service owns its data — no shared database.
🚪
API Gateway / BFF
Single entry point for routing, auth, rate-limiting, aggregation. A Backend-for-Frontend tailors responses per client (web/mobile).
🔎
Service discovery
Services find each other dynamically (DNS, Consul, Kubernetes Services) rather than hard-coded IPs, so instances can scale and move.
🔭
Observability
Distributed tracing (W3C TraceContext, OpenTelemetry), correlation IDs, centralized logs & metrics — you can't debug what you can't trace across hops.
The theorem everyone quotesCAP & consistency trade-offs

In the presence of a network Partition you must choose between Consistency and Availability. Networks do partition, so the real choice is CP vs AP.

ChoiceBehaviour on partitionExample fit
CP (consistency)Reject/limit requests to avoid stale dataPayments, ledgers, inventory decrement
AP (availability)Keep serving, reconcile later (eventual)Feeds, catalogs, recommendations, carts
💡
PACELC extends CAP: even when there's no partition (E), you trade Latency vs Consistency. Strong consistency costs round-trips/coordination; eventual consistency buys speed. Pick per use-case, not globally.
Sync vs asyncHow services communicate

Synchronous (HTTP/gRPC)

  • Simple request/response, immediate result
  • Temporal coupling — caller blocks on callee being up
  • Failure & latency cascade across the call chain
  • Good for queries needing a fresh answer now

Asynchronous (messaging/events)

  • Producers & consumers decoupled in time
  • Resilient — broker buffers if a consumer is down
  • Natural fit for workflows, fan-out, eventual consistency
  • Harder to reason about; needs idempotency & ordering care
🚫
Distributed transactions (2PC) are an anti-pattern across services — they couple availability and don't scale. Use the Saga pattern (a sequence of local transactions with compensating actions) for cross-service consistency instead. More in the messaging and CQRS sections.
Topic 17
Asynchronous Processing & Event Messaging
Queues and event streams are how distributed systems stay decoupled and resilient. Know the delivery guarantees and the failure modes.
📨
Queue (point-to-point)
One message → one consumer in a competing-consumers group. Distributes work & load-levels spikes. e.g. Azure Service Bus queue, SQS, RabbitMQ.
📢
Topic (pub/sub)
One event → many independent subscribers, each with its own copy. Enables fan-out & loose coupling. e.g. Service Bus topics, SNS, Kafka topics.
📜
Event log / stream
An append-only, replayable log (Kafka, Event Hubs). Consumers track their own offset; events are retained, not deleted on read.
☠️
Dead-letter queue
Where poison messages go after max retries — so one bad message can't block the queue. Always have a DLQ + a replay/inspection plan.

The dual-write problem: saving to your DB and publishing to a broker in two separate steps can fail in between → inconsistency. Fix with the Transactional Outbox: write the event to an outbox table in the same DB transaction, then a relay publishes it reliably. Consumers dedupe on the event id.
Transactional outbox — atomic state change + event
using var tx = await _db.Database.BeginTransactionAsync(ct); _db.Orders.Add(order); _db.Outbox.Add(new OutboxMessage // same transaction { Type = "OrderPlaced", Payload = JsonSerializer.Serialize(evt), Id = Guid.NewGuid() }); await _db.SaveChangesAsync(ct); await tx.CommitAsync(ct); // order + event commit together, or neither // a background relay reads Outbox → publishes to the broker → marks sent
Cross-service workflowsThe Saga pattern

A saga splits a business transaction into local transactions, each emitting an event that triggers the next step. If a step fails, run compensating actions to undo prior steps — no global lock.

Choreography

  • Services react to each other's events — no central coordinator
  • Loosely coupled, easy to add steps
  • Harder to see the whole flow / debug

Orchestration

  • A central orchestrator drives each step explicitly
  • Flow is visible & testable in one place
  • Orchestrator can become a coupling hotspot
🔢
Ordering & backpressure: only guarantee order where you must (partition by an entity key so a given order's events stay sequential). Use prefetch limits / consumer concurrency and let the broker buffer to apply backpressure rather than overwhelming a slow consumer. In .NET, MassTransit or NServiceBus give you retries, DLQ, sagas, and outbox out of the box over RabbitMQ/Azure Service Bus.
Topic 18
Distributed Caching — Redis Patterns & Pitfalls
An in-memory cache (IMemoryCache) dies with the instance and isn't shared. Scale-out needs a distributed cache (Redis) — with its own failure modes.
IMemoryCacheIDistributedCache (Redis)
ScopePer-process, per-instanceShared across all instances
Survives restartNoYes (separate process/cluster)
SpeedFastest (no serialization/network)Fast, but network + serialize cost
Use forTiny hot data on one nodeShared sessions, multi-instance read cache
↩️
Cache-aside
App checks cache → on miss reads DB → populates cache. The default, most flexible pattern. App owns invalidation.
✍️
Write-through / behind
Through: write cache + DB together (consistent, slower writes). Behind: write cache now, DB async (fast, risk of loss).
⏲️
TTL & eviction
Every key needs a TTL. When full, Redis evicts by policy (LRU/LFU). Absolute + sliding expiration balance freshness vs hit rate.
🧯
Invalidation
"There are only two hard things…" Evict/refresh on writes, key by version/tag, or publish invalidation events so all nodes drop stale entries.
🌩️
The three killers:
• Stampede / dog-pile — a hot key expires and thousands of requests hit the DB at once. Fix with a lock/single-flight or probabilistic early refresh.
• Penetration — repeated misses for keys that don't exist hammer the DB. Cache a null sentinel or use a Bloom filter.
• Avalanche — many keys expire simultaneously. Fix by jittering TTLs.
Stampede protection — single-flight with a lock
private readonly SemaphoreSlim _lock = new(1, 1); public async Task<Product> GetAsync(int id, CancellationToken ct) { var key = $"product:{id}"; var hit = await _cache.GetStringAsync(key, ct); if (hit is not null) return Deserialize(hit); // fast path await _lock.WaitAsync(ct); // only ONE caller rebuilds try { hit = await _cache.GetStringAsync(key, ct); // double-check after lock if (hit is not null) return Deserialize(hit); var product = await _db.Products.FindAsync([id], ct); await _cache.SetStringAsync(key, Serialize(product), new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) + Jitter() }, ct); // jitter avoids avalanche return product; } finally { _lock.Release(); } }
⚖️
Cache only what helps: read-heavy, expensive-to-compute, tolerant-of-slight-staleness data. Don't cache rapidly-changing or per-user-unique data blindly. Always measure hit ratio — a low-hit cache adds latency and complexity for nothing. Redis also does more than caching: distributed locks, pub/sub, rate limiting, leaderboards.
Topic 19
Domain-Driven Design (DDD)
DDD aligns the model with the business and gives you the language to draw service boundaries. It's about taming complexity in the core domain — not ceremony for CRUD.
🗺️
Bounded Context
An explicit boundary within which a model & its ubiquitous language are consistent. "Customer" can mean different things in Sales vs Support — separate contexts. Often = one microservice.
🧱
Aggregate & Root
A cluster of objects treated as one consistency unit. The aggregate root is the only entry point; it enforces invariants. Reference other aggregates by id, not by object.
🆔
Entity vs Value Object
Entity has identity & lifecycle (an Order). Value object is defined by its attributes, immutable, no id (Money, Address) — record fits perfectly.
📣
Domain Event
Something meaningful that happened in the domain (OrderShipped). Raised by aggregates, handled in-process or published outward to integrate other contexts.
🛡️
The aggregate is your consistency & transaction boundary. One transaction should modify one aggregate; coordinate changes across aggregates with domain events / eventual consistency. Keep aggregates small — large ones cause contention and big loads.
Aggregate root protecting its invariants
public class Order // aggregate root { private readonly List<OrderLine> _lines = new(); public IReadOnlyList<OrderLine> Lines => _lines; // no external mutation public OrderStatus Status { get; private set; } public void AddLine(Product p, int qty) { if (Status != OrderStatus.Draft) // invariant enforced here throw new DomainException("Cannot modify a submitted order"); _lines.Add(new OrderLine(p.Id, qty, p.Price)); } public void Submit() { if (_lines.Count == 0) throw new DomainException("Empty order"); Status = OrderStatus.Submitted; _events.Add(new OrderSubmitted(Id)); // raise domain event } }
🎯
Don't DDD everything. Reserve tactical DDD (aggregates, value objects, domain events) for the complex core domain where business rules live. Generic/supporting subdomains (CRUD admin, lookups) are fine as plain services. The strategic tools (bounded contexts, context mapping, ubiquitous language) are valuable everywhere.
Topic 20
CQRS & Event Sourcing
CQRS separates the model that changes data from the model that reads it. Event sourcing stores the history of changes instead of just the current state. They're independent — use either, both, or neither.
CQRS — separate write & read paths
flowchart LR
  UI(("Client")) -->|Command| CH["Command Handler\n(validate + domain)"]
  CH --> WDB[("Write Model\n(normalized)")]
  WDB -->|events / projection| RDB[("Read Model\n(denormalized)")]
  UI -->|Query| QH["Query Handler"]
  QH --> RDB

Command side (writes)

  • Validates & enforces business rules via aggregates
  • Normalized, consistency-focused model
  • Returns success/failure, not data

Query side (reads)

  • Denormalized, shaped for the screen — fast reads
  • Can use a different store (SQL→Redis/Elastic)
  • Often eventually consistent with the write side
⚖️
When CQRS earns its keep: read & write loads differ wildly, complex domains where the query shape ≠ the domain shape, or you need independently scalable/optimized read stores. The cost: two models, sync between them, and eventual consistency the UI must handle. For simple CRUD it's over-engineering — a shared model is fine. You can also do "lite" CQRS: separate command/query objects (MediatR) over the same database.
Lite CQRS with MediatR — split command & query
// Command — changes state, returns no data public record PlaceOrder(int CustomerId, List<LineDto> Lines) : IRequest<Guid>; public class PlaceOrderHandler : IRequestHandler<PlaceOrder, Guid> { public async Task<Guid> Handle(PlaceOrder cmd, CancellationToken ct) { ... } } // Query — returns a read-optimized DTO, no domain logic public record GetOrderSummary(Guid Id) : IRequest<OrderSummaryVm>;
Storing history, not stateEvent Sourcing

Instead of UPDATE-ing a row, you append immutable events (OrderPlaced, ItemAdded, OrderShipped). Current state is rebuilt by replaying events. The event log is the source of truth.

What you gain

  • Full audit trail & time-travel (state at any point)
  • Natural fit for events/integration & CQRS projections
  • No lossy updates — every change is a fact

What it costs

  • Higher complexity; querying needs projections
  • Schema/versioning of events over time is hard
  • Replays get slow → need snapshots
  • Eventual consistency; steeper learning curve
🧨
Don't reach for event sourcing by default. It shines for audit-critical, behavior-rich domains (finance, logistics, collaboration). For most apps it's accidental complexity — plain CRUD or lite CQRS is the right call. CQRS does not require event sourcing, and vice-versa.
Topic 21
Large Data & Streaming — Don't Buffer the World
The cardinal rule for big data in .NET: stream it, don't load it all into memory. Buffering a million rows or a multi-GB file is how you get OutOfMemoryException and LOH fragmentation.

Buffering (dangerous at scale)

  • ToList() on a huge query → all rows in RAM
  • ReadAllBytes on a big file
  • Materializing before transforming/writing
  • Memory grows with data size → OOM, GC pauses

Streaming (constant memory)

  • IAsyncEnumerable<T> + await foreach
  • Stream pipelines, PipeReader/PipeWriter
  • Process & release each item/chunk as it arrives
  • Memory stays flat regardless of total size
Buffers everything — OOM risk on large tables
var all = await _db.Events.ToListAsync(); // millions of rows in memory foreach (var e in all) Export(e);
Streams row-by-row — flat memory
await foreach (var e in _db.Events .AsNoTracking() .AsAsyncEnumerable().WithCancellation(ct)) // one row at a time { await ExportAsync(e); // processed & released }
Producer/consumer in-processSystem.Threading.Channels

Channel<T> is the modern .NET primitive for async producer/consumer pipelines with built-in backpressure — a bounded channel makes fast producers wait when consumers fall behind.

Bounded channel — backpressure between producer & consumer
var channel = Channel.CreateBounded<Record>(1000); // cap = backpressure // Producer _ = Task.Run(async () => { await foreach (var r in ReadSourceAsync(ct)) await channel.Writer.WriteAsync(r, ct); // blocks when full channel.Writer.Complete(); }); // Consumer(s) await foreach (var r in channel.Reader.ReadAllAsync(ct)) await ProcessAsync(r);
🌊
Batch vs Stream
Batch: bounded data, run periodically (ETL, reports). Stream: unbounded, continuous, low-latency (Kafka, Event Hubs, Flink). Pick by latency & data boundedness.
🚰
Backpressure
When consumers can't keep up: bound buffers, throttle producers, or shed/sample load. Never let an unbounded queue grow until OOM.
🧱
Chunking & paging
Process big sources in chunks/keyset pages. For files, read fixed-size buffers; for DBs, paginate by an indexed key, not OFFSET on huge tables.
🔀
Parallel pipelines
Parallel.ForEachAsync or TPL Dataflow for bounded-concurrency CPU/IO stages with throttling & ordering control.
📡
End-to-end streaming: stream from the DB (AsAsyncEnumerable), keep it lazy through transforms (yield return), and stream to the client (IAsyncEnumerable from a minimal API / gRPC server streaming, or write directly to the Response.Body). Bytes flow through with bounded memory the whole way. Compress (gzip/brotli) and use server streaming for very large responses.
Part B
Interview Questions — Sharp Answers
Concise, senior-level responses with the trade-offs interviewers are listening for.
Question 1If you use .AsEnumerable() in a repository and have other queries to run, what's the problem?

Calling AsEnumerable() (or ToList()) forces immediate execution and switches from IQueryable to IEnumerable. From that point on, everything runs in memory, not in SQL.

What breaks

  • The whole table is pulled into RAM before any further filtering
  • Later Where/OrderBy/Skip/Take no longer translate to SQL → no DB indexes used
  • Paging happens after fetching everything → huge over-fetch
  • Memory pressure & GC churn; can starve the connection pool

Do instead

  • Keep the chain as IQueryable until the very end
  • Compose filters, then call ToListAsync() once
  • Only materialize the rows you actually need
Problem
var q = _context.Orders.AsEnumerable(); // loads ALL orders now var recent = q.Where(o => o.Date > cutoff) // in-memory, no index .Take(20);
Fix
var recent = await _context.Orders .Where(o => o.Date > cutoff) // WHERE in SQL, uses index .OrderByDescending(o => o.Date) .Take(20) .AsNoTracking() .ToListAsync();

💥
Can you handle exceptions in async void? Only if you wrap the entire body in try/catch — an unhandled exception escaping an async void method cannot be caught by the caller and will typically crash the whole app. The trade-offs: not awaitable, hard to test, exceptions are deadly. Use async Task everywhere except UI event handlers.
Acceptable async void — UI handler with full try/catch
private async void OnSave_Click(object s, EventArgs e) { try { await _service.SaveAsync(); } catch (Exception ex) { ShowError(ex); } // must catch here }

Question 4What's the problem injecting Scoped services into a Singleton, and how do you fix it?

The captive dependency problem. The singleton lives for the whole app, so the scoped instance it captures also lives forever — defeating the per-request lifetime.

Symptoms

  • A per-request DbContext shared across all requests
  • DbContext is not thread-safe → race conditions, corrupted data
  • ObjectDisposedException once the first scope ends
  • Stale data; the scoped service never refreshes

Fixes

  • Inject IServiceScopeFactory and create a scope per unit of work
  • Or inject a factory / Func<T> instead of the service itself
  • Or make the consumer Scoped instead of Singleton
  • Enable scope validation (on by default in Development) to catch it at startup
Fix — resolve the scoped service inside an explicit scope
public class BackgroundWorker // registered as Singleton { private readonly IServiceScopeFactory _scopeFactory; public BackgroundWorker(IServiceScopeFactory f) => _scopeFactory = f; public async Task DoWork() { using var scope = _scopeFactory.CreateScope(); var db = scope.ServiceProvider .GetRequiredService<AppDbContext>(); // fresh per scope await db.SaveChangesAsync(); } }

Heuristic: "Program to an interface" — favour interfaces for contracts/DI/testability. Reach for an abstract class only when concrete subtypes genuinely share state and base implementation. Often you use both: an interface for the contract, an abstract base for shared plumbing.
Question 6Do you store images/files in S3/Blob or in a managed database? Which & why?

Object storage (S3 / Azure Blob), not the database. Store the binary in the blob store; persist only the metadata and the key/URL in the DB.

1
Cost & scale: object storage is far cheaper per GB and effectively unlimited; DB storage is premium.
2
Performance: large BLOBs bloat DB backups, hurt buffer-cache hit ratios, and slow every query/replication.
3
Delivery: blob stores serve via CDN and presigned URLs, offloading bandwidth from your app servers.
4
Durability: S3/Blob give 11 nines of durability, lifecycle tiering, and versioning out of the box.
✅
Exception: very small files (< a few KB, e.g. tiny thumbnails) or when strict transactional consistency with the row is mandatory — then storing in the DB can be justified. Otherwise: blob + metadata row.

The quick-hit questions interviewers use to probe depth. One crisp distinction each.

QuestionSharp answer
const vs readonlyconst is compile-time, baked into callers (versioning hazard across assemblies). readonly is set at runtime in the constructor — use it for values that depend on construction.
== vs .Equals()For reference types == is identity by default; .Equals() can be overridden for value semantics. string overloads both to compare content. Records give value equality for free.
Task vs ValueTaskUse ValueTask for hot paths that often complete synchronously (avoids an allocation). Never await a ValueTask twice or store it — it's single-consumption.
ConfigureAwait(false)In library code, skip capturing the SynchronizationContext to avoid deadlocks and reduce overhead. Not needed in ASP.NET Core (no context), still good practice in shared libraries.
IEnumerable vs ICollection vs IListIncreasing capability: IEnumerable (iterate) → ICollection (count/add/remove) → IList (indexed). Expose the narrowest one a caller needs.
first vs FirstOrDefaultFirst throws if empty; FirstOrDefault returns the default. Single additionally asserts exactly one — use it to encode an invariant.
yield returnBuilds a lazy iterator (state machine) — items are produced on demand, enabling streaming and infinite sequences without buffering.
Covariance / contravarianceout T (covariant, IEnumerable<out T>) lets you use a more-derived type; in T (contravariant, IComparer<in T>) a less-derived one.
Dispose vs finalizerDispose is deterministic cleanup you call (via using); a finalizer is a non-deterministic GC safety net. Implement the pattern + SuppressFinalize.
Delegate vs eventAn event is a delegate with restricted access — subscribers can only +=/-=, not invoke or overwrite the list. Prefer events (or Action/Func) for the publisher/subscriber boundary.
Closures captureLambdas capture variables, not values. Capturing a loop variable (pre-C# 5 for, or a shared variable) is a classic bug; capture a local copy.
Managed vs unmanagedManaged resources are handled by the GC; unmanaged (file handles, sockets, native memory) must be released explicitly via IDisposable.
throw vs throw exthrow; preserves the original stack trace; throw ex; resets it — almost always wrong.
Stateless vs stateful servicesPrefer stateless services (scale horizontally, no sticky sessions). Externalize state to a DB/cache/Redis so any instance can serve any request.

Production Operations & Incident Handling

Question 7How do you approach production incident troubleshooting? (step by step)
1
Acknowledge & assess impact. Declare the incident, assign a severity, identify who/what is affected.
2
Stop the bleeding first. Mitigate before you diagnose — roll back, scale out, feature-flag off, or failover. Restore service, then investigate.
3
Gather evidence. Logs, metrics, traces, dashboards, recent deploys/config changes. Establish a timeline — "what changed?"
4
Form & test a hypothesis. Narrow the blast radius (which service, which dependency, which release). Reproduce if possible.
5
Apply the fix (hotfix or roll forward) and verify recovery via metrics & synthetic checks.
6
Communicate continuously with stakeholders throughout — status, ETA, resolution.
7
Blameless post-mortem. Document root cause, add monitoring/alerts/tests so it can't recur silently.

Question 9What are AWS ACLs, Resources, and Subnets? How do you enforce security?

Stateless subnet-level firewall with ordered allow/deny rules for inbound & outbound. A coarse perimeter layer.
🪣

Permissions attached to a resource (e.g. S3 bucket policy). Modern practice: prefer IAM policies & bucket policies, keep ACLs disabled / "Block Public Access" on.

Question 10As a senior, how do you handle production crash scenarios?
1
Stay calm & lead. Coordinate the response, assign roles (comms, investigation, mitigation), avoid panicked random changes.
2
Restore service first. Failover, roll back, scale, restart, or shed load. Customer impact over root cause in the moment.
3
Use telemetry, not guesses. APM (App Insights/Datadog), structured logs, distributed traces, dumps for crashes (e.g. dotnet-dump).

5

Question 11Which architecture/patterns do you use? How do you ensure performance?

Architecture & patterns

  • Clean / Layered (or Hexagonal) for separation of concerns
  • Repository + Unit of Work where it adds value (not over-abstracted)
  • CQRS + MediatR for complex read/write paths
  • Dependency Injection everywhere; SOLID principles
  • Microservices only when the org & domain justify it — modular monolith first
  • Async messaging / outbox for cross-service consistency

Performance optimizations

  • Fix N+1, use AsNoTracking, project to DTOs, index hot columns
  • Async I/O end-to-end; never block on async
  • Caching: in-memory + distributed (Redis), with sensible invalidation
  • Pagination & server-side filtering — never return everything

Part D — Practical Task
Bug Fix — Orders Showing $0 & Recent Orders Missing
Two reported symptoms, one piece of code. Let's find the root causes and fix them.
🐞
Reported symptoms: (1) order total always shows $0, and (2) recent orders don't populate for certain customers.
Original pseudocode (as given)
for (var customer in customers) { customer.Orders = _context.Orders.Where(o => o.customerid == customer.id); for (var order in customer.Orders) { lineitem.Amount = order.LineItems?.TotalAmount(x => x.sum(amount)) ?? 0; } return view(customer); }
Root cause analysisWhat is actually wrong
#BugEffect
1 order.LineItems is never loaded (no Include). With lazy loading off it's null → ?? 0 hits every time. Total always $0
2 The sum is assigned to a single, outer lineitem.Amount variable (overwritten each loop) instead of accumulating onto the order/customer total. Total always $0 / wrong
3 return view(customer) is inside the loop → it returns after the first customer. Recent orders missing for "certain" customers (everyone after the first)
4 Where(...) returns an un-executed IQueryable; assigning it to customer.Orders then re-querying per customer = the N+1 problem. Slow + inconsistent results
5 No OrderBy/Take for "recent" orders, and no null/empty handling. "Recent" not actually recent
🎯
The two reported symptoms map cleanly: the $0 total comes from bugs #1 & #2 (line items never loaded + overwritten variable). The missing recent orders for certain customers comes from bug #3 (the return sits inside the loop, so only the first customer is ever processed).
The fixCorrected pseudocode
Eager-load the graph once, accumulate correctly, return after the loop
// 1) Load customers + orders + line items in ONE query (no N+1). // AsNoTracking because this is a read-only listing page. var customers = await _context.Customers .Include(c => c.Orders.OrderByDescending(o => o.Date).Take(10)) .ThenInclude(o => o.LineItems) .AsNoTracking() .ToListAsync(ct); foreach (var customer in customers) { foreach (var order in customer.Orders) { // 2) Accumulate onto THIS order, guard nulls/empties. order.TotalAmount = order.LineItems?.Sum(li => li.Amount) ?? 0m; } } // 3) Return AFTER the loop — render ALL customers, not just the first. return View(customers);
✅
Better still: compute the total in the database with a projection to a view-model so no entity mutation is needed and even less data crosses the wire:
Even cleaner — project totals in SQL
var model = await _context.Customers .Select(c => new CustomerVm { Id = c.Id, Name = c.Name, Recent = c.Orders .OrderByDescending(o => o.Date) .Take(10) .Select(o => new OrderVm { Id = o.Id, Date = o.Date, Total = o.LineItems.Sum(li => li.Amount) // SUM() in SQL }).ToList() }) .AsNoTracking() .ToListAsync(ct); return View(model);
Performance follow-upThe customer listing page gets huge traffic — how do you make it fast? Cache?

Yes — but caching is the last layer, not the first. Fix the query, then add caching and other layers in order of cost/benefit.

1
Fix the query first. Eliminate N+1 (Include/projection), AsNoTracking, select only needed columns into a DTO, and index CustomerId + Order.Date.
2
Paginate. Never return all customers — server-side paging with Skip/Take (or keyset pagination for big tables).
3
Cache reads. Distributed cache (Redis) for the listing payload, or IMemoryCache for a single instance. Use short TTLs + cache-aside pattern.
4
Invalidate smartly. Evict/refresh on writes (new order, updated customer) so the cache never goes stale; key by page + filter.
5
Output/response caching & CDN for anonymous or rarely-changing views; ETags for conditional requests.
6
Scale out behind a load balancer; consider read replicas for read-heavy reporting.
Cache-aside with Redis (IDistributedCache)
public async Task<List<CustomerVm>> GetPageAsync(int page, CancellationToken ct) { var key = $"customers:page:{page}"; var cached = await _cache.GetStringAsync(key, ct); if (cached is not null) return JsonSerializer.Deserialize<List<CustomerVm>>(cached)!; // hit var data = await QueryFromDbAsync(page, ct); // miss → DB await _cache.SetStringAsync(key, JsonSerializer.Serialize(data), new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(2) }, ct); return data; }
🧠
Order of operations matters: caching a slow, broken query just serves wrong data faster. Make it correct, then efficient (query + indexes + pagination), then cached, then scaled. Always measure with load tests before and after.
